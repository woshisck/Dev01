DevKit GameplayTag 重整方案

结合当前项目系统梳理、TagLevel.md 和 LOTF 分析报告后的命名结论。本文用于给同事同步：哪些根节点保留、哪些旧根迁移、后续清理按什么顺序执行。

日期：2026-06-23 状态：规范冻结草案 文档：Docs/04_开发实现与系统文档/标签/GameplayTag_ReorgPlan_LOTF.md

一、LOTF 规范简述与评价

LOTF 的核心不是照搬一堆根节点，而是把 Tag 当成工程资产治理：配置字典负责设计师可见 Tag，GameState.Flags 单独承载高频存档/任务 Flag，C++ Native Tag 负责代码安全访问，所有重命名必须通过 Redirect 保护旧资产和存档。
优点

权威来源清晰、重命名安全、C++ 查询稳定、UI/状态/存档职责分开。

风险

如果职责表不清楚，根节点会越来越多，最后变成“什么都能放”的 Tag 池。

DevKit 采用方式

采用治理方式，不照抄根节点；按 DevKit 的角色、Buff、Rune、Director、Level 等系统重新划根。

三、关键结论

Buff 独立

Buff.* 是一级大系统。Buff 阻断动作时，StateConflict 直接读 Buff.Status.*，不复制到 Character.State.*。

状态不按玩家/敌人拆

死亡、受击、击退、技能执行中都属于通用角色状态，统一使用 Character.State.*，减少 Block 规则和资产 Tag 数量。

Rune/Card 合并

正式根统一为 Rune.*。旧 Card.* 和 Combo.CombatDeck.* 是迁移来源；正式命名不用 Rune.Deck.*，改用 Binding / Activation / Trigger / FlowRole。

Director 主系统

Story/Tutorial 提升为 Director.*，可覆盖房间、掉落、任务、对话、教程步骤；Save 只记录结果。

UI 只表现

HUD Buff 图标来自 ASC 的 ActiveGameplayEffect / GameplayTag，经 ViewModel 或事件显示，不在 UI 中保存业务状态。

重命名走 Redirect

新增正式 Tag、添加 Redirect、迁移资产、确认无引用后再删除旧 Tag。不能直接硬替换。

四、核心迁移映射

PlayerState.AbilityCast.Attack      -> Character.State.Skill.Attack
PlayerState.AbilityCast.WeaponSkill -> Character.State.Skill.WeaponSkill
PlayerState.AbilityCast.Skill       -> Character.State.Skill.Active
PlayerState.AbilityCast.Dash        -> Character.State.Movement.Dash

Card.ID.*                           -> Rune.ID.*
Card.Effect.*                       -> Rune.Effect.*
Combo.CombatDeck.ActionSlot.*       -> Rune.Binding.Action.*
Combo.CombatDeck.FlowRole.*         -> Rune.FlowRole.*
Combo.TriggerTiming.*               -> Rune.Trigger.*

五、后续执行顺序

  1. 冻结本规范，后续新增 Tag 按本文执行。
  2. 扫描当前 Config/Tags，生成根节点清单和旧到新映射表。
  3. 把 C++ 硬编码 Tag 字符串收口到统一访问层。
  4. 新增正式 Tag 和 GameplayTagRedirects。
  5. 迁移 GA、AbilityData、RuneDataAsset、CombatDeck、BuffFlow、GE、UI 映射资产。
  6. 保留旧 QTE Finisher 资产但不启用运行时入口。
  7. 先做静态扫描验证；Unreal 编译/编辑器重保存只在明确要求时执行。
注意：Rune.FlowRole.Finisher 可以表示普通组合终结位，但不能重新接回已废弃的 QTE Finisher 系统。
