一、推荐系统总览

当前项目不是传统“角色系统、武器系统、敌人系统”平铺结构，而是“GameInstance/World 生命周期层 → Character Actor 宿主层 → GAS/组件/DA 驱动玩法层”。清理时建议先按职责边界分系统，再按运行时归属决定 Tag 根。

二、全量大系统清单

01 CoreRuntime / 游戏生命周期

保留主系统

管理单局和跨局生命周期，是关卡、剧情、存档、局外成长的上层宿主。

  • GameInstance 层：Save、Story、Tutorial、MetaProgression。
  • World/GameMode 层：房间阶段、敌人注册、奖励生成、传送门。
  • 不建议把具体战斗行为塞进 CoreRuntime。
YogGameMode.*YogGameInstanceBase.*YogWorldSubsystem.*

02 Character / 角色 3C 宿主

保留主系统

角色不是单纯属性容器，而是输入、移动、相机、ASC、装备、组件的运行时宿主。

  • CharacterBase：ASC、Base/Damage/Rune AttributeSet、交互、输入缓存。
  • PlayerCharacter：背包、卡组、主动技能、献祭、武器双槽。
  • EnemyCharacter：敌人武器、AI 上下文、血条、BuffFlow。
YogCharacterBase.*PlayerCharacterBase.*EnemyCharacterBase.*

03 Combat / GAS 战斗

保留主系统

当前玩家运行时战斗主链是 Controller 输入 → PlayerState.AbilityCast.* → GA → AbilityData 蒙太奇 → Notify/伤害/卡组。

  • 四个独立动作：Attack、Skill、WeaponSkill、Dash。
  • 通用反应：HitReact、Knockback、Dead、SuperArmor、HitStop。
  • ComboGraph 对玩家运行时已不应作为主路径。
GA_PlayerAttackCombos.*GA_WeaponSkill.*GA_PlayerDash.*AbilityData.*

04 Weapon & Equipment / 武器装备

保留主系统

武器是战斗数据和装备表现的来源，不直接拥有玩家输入逻辑。

  • WeaponDefinition 提供 AttackAbilityData、WeaponSkillAbilityData、PassiveAbilityData。
  • 装备时挂 Weapon.Type.* LooseTag，驱动近战/远程 GA 路径隔离。
  • 武器还提供动画层、展示数据、初始 CombatDeck。
WeaponDefinition.*WeaponInstance.*WeaponSpawner.*Config/Tags/Equip.ini

05 Interact & Item / 交互物与道具

待清理确认

当前散落在 Character、Map、Item、Actors、Component 中，建议统一归为场景交互域。

  • 交互组件：PropInteractComponent、PlayerInteraction、InteractPrompt。
  • 交互物：武器拾取、Portal、Teleporter、Shop、Altar、RewardPickup。
  • 战斗道具：CombatItemComponent、CombatItemAreaActor。
PlayerInteraction.*PropInteractComponnet.*CombatItemComponent.*Map/*

06 Enemy & AI / 敌人与 AI

保留主系统

敌人系统单独存在；Tag 管理上建议把“通用角色状态”放到 Character.State，把“敌人专属决策/技能身份”放到 Enemy.*。

  • 通用状态：死亡、受击、击退、眩晕等可共用 Character.State.*。
  • AI 状态：Patrol、Alert、Combat，建议使用 Enemy.AI.State.* 或黑板枚举，不放 Character.State。
  • 攻击选择：AttackProfile 按 Range、Weight、Cooldown、Role 选招。
  • 敌人武器使用 EnemyWeaponDefinition，和玩家 WeaponDefinition 分开。
EnemyData.*YogAIController.*BTTask_EnemyAttackByProfile.*MobSpawner.*

07 Rune/Card/BuffFlow / 符文卡组

保留主系统

这是项目最核心的构筑系统之一，包含战斗卡组、背包热度、BuffFlow 节点图和符文 DA。

  • CombatDeck 按 ActionSlot：Attack、Skill、WeaponSkill、Dash 解析卡。
  • RuneDataAsset 描述符文身份、触发、连携、战斗卡配置。
  • BuffFlow 是符文和 Buff 的可视化执行引擎。
CombatDeckComponent.*RuneDataAsset.*BuffFlow/*BackpackGridComponent.*

08 Level/RunLoop/Reward / 关卡循环

保留主系统

负责房间选择、刷怪、结算、奖励和切关，是 Roguelite 单局循环骨架。

  • Room.Type：Normal、Elite、Shop、Event。
  • Room.Layer：L1、L2、L3。
  • LevelFlow 可用于关卡内脚本、教程触发和演出。
RoomDataAsset.*CampaignDataAsset.*LevelFlow/*RewardPickup.*

09 Director / Story / Tutorial / 导演剧情教程

保留主系统

建议提升为“导演系统”：它不拥有战斗/关卡/UI 的底层实现，而是通过接口编排和覆盖其他系统。

  • 可覆盖下一房间、固定掉落、特殊敌人、传送门和首轮教程节奏。
  • 可下发任务目标：限时击杀、护送、到达指定点、触发对话。
  • 可调用 Character/AI/UI 接口：移动到点位、锁输入、开对话框、显示教程。
StoryEngineSubsystem.*StoryEncounterRuntimeSubsystem.*FirstRunTutorialDirectorSubsystem.*TutorialManager.*

10 UI/HUD/UX / 界面

保留主系统

HUD 负责表现和玩家操作入口；玩家/敌人的 Buff 图标应由状态源驱动，UI 只订阅和展示，不决定状态。

  • HUDRoot：血条、敌人箭头、武器/卡组/技能/战斗物品栏。
  • Buff 图标建议链路：ASC GameplayTag/ActiveGameplayEffect → UI ViewModel/事件 → HUD Icon List。
  • Overlay：背包、暂停、LootSelection、教程、PortalPreview。
  • 编辑器 UI 与运行时 UI 应分开命名。
YogHUD.*YogHUDRootWidget.*UI/*Widget.*

11 Save & Progression / 存档局外成长

保留主系统

负责跨关和跨局保存，包含 HP、金币、武器、卡组、符文、教程进度、局外升级；教程流程由导演系统决定，Save 只记录结果。

  • SaveSubsystem 是读写入口。
  • MetaProgression 管理局外货币、节点解锁、功能解锁。
  • RunState 与 PlayerCharacter 的恢复逻辑强相关。
YogSaveSubsystem.*YogSaveGame.*YogMetaProgressionSubsystem.*

12 Presentation / 相机表现

保留主系统

包括相机、视觉反馈、材质、Niagara、水体和动作表现辅助。

  • Camera：CameraPawn、CameraComponent、CameraVolume、遮挡淡出。
  • VFX：CharacterFlash、GameplayCue、Niagara、时间缩放。
  • Animation Notify 属于战斗/表现之间的桥。
Camera/*Animation/*Visual/*Water/*

13 EditorTools / 编辑器自动化

保留主系统

非运行时系统，但对资产生产和迁移很重要，应从 Runtime 命名中独立。

  • Commandlet：战斗蒙太奇同步、敌人模板、UI 设置、故事导入。
  • Workbench：DataEditor、RuneEditor、LevelData、MetaProgression、StoryEvent。
  • 建议统一前缀 Editor.* 或 Tools.*，不要混进 GameplayTag 运行时根。
Source/DevKitEditor/*Tools/*

14 LegacyCompatibility / 旧兼容

废弃兼容

为了老资产加载保留，但不应继续作为新系统入口；本轮已检查运行时入口并收紧 TemporaryFinisher 旧开关。

  • ComboGraph：玩家运行时不再走主路径。
  • Special/SpecialAttack：被 ActiveSkill 和 WeaponSkill 取代。
  • Finisher QTE：native 代码已移除，Tag/资产仅为兼容。
  • 已隔离：GameMode 的 TemporaryFinisher 计数在 Finisher 废弃时不会运行。

五、对你示例 Tag 的修正规格

角色 / 战斗动作

旧拼写 Skil 建议修为 Skill。若用于 Block/Cancel，就放在 State；若用于查找/激活 GA，就放在 Skill。

  • Character.State.Skill.Attack
  • Character.State.Skill.Active
  • Character.State.Skill.WeaponSkill
  • Character.State.Movement.Dash
  • Character.Skill.Attack
  • Character.Skill.WeaponSkill

交互 / 武器拾取

建议统一 PascalCase，不混用 melee/Melee；Interact 表示场景交互，Weapon 表示装备后运行态。

  • Interact.Weapon.Melee
  • Interact.Weapon.Melee.SingleHandBlunt
  • Interact.Weapon.Ranged.Bow
  • Weapon.Type.Melee.SingleHandBlunt
  • Weapon.Type.Ranged.Bow

六、优先清理建议

优先级清理项原因建议动作
P0明确废弃兼容边界ComboGraph、Special、Finisher 仍留 Tag/资产，容易被误接回主流程。文档和资产命名加 Legacy/Deprecated；新资产禁止引用。
P0冻结玩家战斗主链当前主链已改为 Attack/Skill/WeaponSkill/Dash，清理时不能回到 Light/Heavy/Special。以 PlayerState.AbilityCast.* 为当前真实源，迁移前不重命名。
P1整理 Interact 系统武器拾取、奖励、商店、祭坛、传送门、战斗道具现在分散。建立 Interact 分类表和 Tag 根，只给场景可交互对象使用。
P1收束 Character.State / Buff.Status / State三者都像“状态”，最容易混乱。Character.State 表示角色自身状态；Buff.Status 表示效果施加状态；State 表示跨系统冲突抽象状态。
P2武器细分类型当前只有 Melee/Ranged，不足以表达单手钝器、弓、火枪等。扩展 Weapon.Type.*，并决定是否同步 Interact.Weapon.*。
P2Docs 与 Config 对齐已有 GameplayTag_MasterGuide 与实际新增 Tag 有偏差。迁移时同步更新 Tag 规范、Config/Tags、DA 制作手册。
