# 项目状态总览

> 用途：让策划、制作、程序和 AI 助手快速判断“这是需求、已完成功能、待配置内容、待开发任务还是待验收项”。  
> 真源说明：任务真源仍是 [TASKS](PM/TASKS.md)，完成功能真源仍是 [FeatureLog](FeatureLog.md)，本页是面向协作的状态看板。

## 状态流转

```mermaid
flowchart LR
  A["策划/设计需求"] --> B["开发方案"]
  B --> C["程序实现"]
  C --> D["策划/TA 配置"]
  D --> E["联调验收"]
  E --> F["完成功能记录"]
  F --> G["长期系统文档/配置手册"]
```

## 怎么读状态

| 状态 | 含义 | 常见负责人 | 应看文档 |
| --- | --- | --- | --- |
| 需求 | 来自策划、设计或版本目标，描述“要做什么/为什么做” | 策划 / 设计 | [01_策划需求与版本方案](../01_策划需求与版本方案/README.md)、[02_玩法设计与调研](../02_玩法设计与调研/README.md) |
| 开发中 | 已拆成接口、组件、任务，但尚未完全落地 | 程序 / AI | [TASKS](PM/TASKS.md)、[AI任务交接模板](../06_AI协作与VibeCoding/AI任务交接模板.md) |
| 程序完成 | C++/脚本/框架已完成，可能仍需要编辑器配置 | 程序 / AI | [FeatureLog](FeatureLog.md)、对应系统文档 |
| 待策划配置 | 代码已具备入口，需要在编辑器内创建/填写 DA、WBP、GA、Montage、Niagara、Tag 等 | 策划 / TA / 技术策划 | [03_策划配置与制作手册](../03_策划配置与制作手册/README.md)、[TASKS P0/P1](PM/TASKS.md) |

## 完成功能应该看哪里

| 想确认 | 看哪里 | 说明 |
| --- | --- | --- |
| 某功能是否完成 | [FeatureLog](FeatureLog.md) | 每条包含状态、涉及文件、配置入口、已知限制 |
| 当前任务是否还需要人做 | [TASKS](PM/TASKS.md) | `执行方=用户` 通常表示策划/TA/编辑器配置 |
| 完成功能是否已沉淀成长期依据 | [04_开发实现与系统文档](../04_开发实现与系统文档/README.md) | 稳定系统文档入口 |
| 完成功能是否还需要配置手册 | [03_策划配置与制作手册](../03_策划配置与制作手册/README.md) | 给策划/TA 的操作入口 |
| 历史方案为什么不用了 | [99_归档](../99_归档/README.md) 或 [98_废弃](../98_废弃/README.md) | 只追溯，不作为当前依据 |

## 新增任务时必须填的四个字段

| 字段 | 示例 | 目的 |
| --- | --- | --- |
| 需求来源 | 策划、玩家反馈、商业计划书、版本目标 | 判断这是“为什么做” |
| 程序状态 | 未开始 / 开发中 / C++ 完成 / 已编译 / 已接入 | 判断是否还需要代码 |
| 配置状态 | 无需配置 / 待 DA / 待 WBP / 待 Montage / 待关卡摆放 | 判断策划/TA 要做什么 |
| 验收方式 | 编译 / 自动化 / PIE / 包体验证 / 策划体验确认 | 判断怎么证明完成 |

新增任务若缺少这些字段，应先补到 [TASKS](PM/TASKS.md) 或对应开发方案中，再进入实现。
