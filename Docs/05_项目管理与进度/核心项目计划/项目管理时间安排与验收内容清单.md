# 核心方案项目管理时间安排与验收内容清单

## 目标

本文件用于管理 核心方案从开发启动到版本验收的时间安排、交付物、负责人、验收标准和风险处理。

核心方案的验收目标是跑通以下最小闭环：

```text
进入房间 -> 看到 1D 卡组 -> 轻重攻击消耗卡牌 -> 卡组打空 -> 洗牌空窗 -> 卡组回填 -> 清场奖励 -> 奖励加入卡组 -> 下一房间验证
```

## 阶段验收清单

### 阶段 0：文档与接口冻结

| 验收项 | 标准 |
| --- | --- |
| 系统方向冻结 | 明确 1D 攻击卡组为前台主系统，旧 2D 背包为后台系统 |
| 分组冻结 | 明确背包符文卡组、战斗、关卡、UI、热度 BuffFlow、联调验收 6 个 Codex 小组 |
| 接口冻结 | `ResolveAttackCard(ActionType, bIsComboFinisher, bFromDashSave)` 不再随意改名或改语义 |
| 验收主线冻结 | 最小战斗循环可以被所有小组共同理解 |

### 阶段 1：卡组底座开发

| 验收项 | 标准 |
| --- | --- |
| 卡牌数据结构 | 支持 CardType、RequiredAction、BaseFlow、MatchedFlow、LinkMode、FinisherRequirement |
| 卡组运行时 | 支持 DeckList、ActiveSequence、CurrentIndex、DeckState、ShuffleCooldown |
| 固定顺序 | 每轮洗牌后顺序一致，不随机 |
| 卡牌消耗 | 每次攻击事件最多消耗 1 张卡 |
| 动作不匹配 | 卡牌仍消耗，只触发 BaseFlow |
| 洗牌空窗 | 卡组打空后 UI 进入空状态，攻击不触发卡牌 |

### 阶段 2：战斗事件接入

| 验收项 | 标准 |
| --- | --- |
| Light 事件 | 轻攻击调用卡组接口并传 Light |
| Heavy 事件 | 重攻击调用卡组接口并传 Heavy |
| 终结击事件 | 连招最后一击传 `bIsComboFinisher = true` |
| DashSave 事件 | DashSave 后攻击传 `bFromDashSave = true` |
| 攻击不被阻止 | 洗牌期间攻击照常播放和造成基础伤害 |

### 阶段 3：UI 与表现接入

| 验收项 | 标准 |
| --- | --- |
| 开局卡组显示 | 进入房间后能看到当前 1D 卡组 |
| 消耗反馈 | 每次卡牌消耗后 UI 同步变化 |
| 洗牌反馈 | 卡组打空后 UI 清空或显示装填进度 |
| 回填反馈 | 洗牌结束后 UI 按固定顺序回填 |
| 触发原因 | 连携、终结、热度强化有明确提示 |

### 阶段 4：关卡奖励接入

| 验收项 | 标准 |
| --- | --- |
| 小怪掉落 | 小怪掉金币或小资源，不打断战斗节奏 |
| 清场主奖励 | 清场后出现三选一奖励 |
| 奖励入背包 | 选择奖励后写入后台背包 |
| 奖励入卡组 | 选择奖励后更新 DeckList |
| 不开旧背包 | 选择奖励后不自动打开旧 2D 背包 UI |
| 可跳过奖励 | 玩家可以直接进门，但放弃该次成长 |

### 阶段 5：热度与 BuffFlow 调整

| 验收项 | 标准 |
| --- | --- |
| 热度新定位 | 热度不再控制符文激活区 |
| 基础效果 | 热度不足不会阻止 BaseFlow 生效 |
| 匹配奖励 | 热度可以强化 MatchedFlow 的倍率或表现 |
| 连携强化 | 连携成功能获得或利用热度收益 |
| 终结强化 | 高热度下终结技表现更强 |

### 阶段 6-9：联调、修复、锁定、验收

| 验收项 | 标准 |
| --- | --- |
| 4 卡打空 | 连续攻击 4 次后进入洗牌 |
| 洗牌期间攻击 | 攻击照常进行，不触发卡牌 |
| 洗牌回填 | 冷却结束后卡组固定顺序回填 |
| Finisher | 只有 Combo Finisher 能触发终结技卡牌额外效果 |
| DashSave | DashSave 不重置卡组 Index，不丢连携上下文 |
| 清场奖励 | 奖励选择后进入后台背包和卡组 |
| 下一房验证 | 下一房间能立刻体验新奖励带来的卡组变化 |

## P0 / P1 / P2 问题分级

| 等级 | 定义 | 处理规则 |
| --- | --- | --- |
| P0 | 阻断最小战斗循环的问题 | 必须在 2026/05/11 内容锁定前修复 |
| P1 | 影响主要体验但有绕过方式的问题 | 512 验收前尽量修复，无法修复则记录为已知问题 |
| P2 | 表现、文案、手感微调问题 | 可进入 512 后续迭代 |

## P0 问题示例

- 玩家攻击无法调用卡组接口。
- 卡组打空后无法恢复。
- 洗牌期间攻击被禁止。
- 奖励选择后无法进入下一关。
- 选择奖励后卡组数据损坏。
- 旧背包 UI 强制打开并阻塞流程。

## 最终版本验收标准

核心方案通过验收需要满足以下条件：

| 类别 | 验收标准 |
| --- | --- |
| 战斗主循环 | 玩家能连续使用 Light / Heavy 攻击触发卡牌 |
| 卡组循环 | 卡牌能消耗、打空、洗牌、回填 |
| 空窗体验 | 洗牌期间玩家仍可攻击，只是不触发卡牌 |
| 构筑成长 | 清场奖励能加入后台背包和卡组 |
| UI 表达 | 玩家能看懂当前卡组、洗牌状态、连携原因 |
| 热度定位 | 热度强化连携和终结技，不再控制激活区 |
| 关卡节奏 | 普通房能验证一轮卡组，精英房能验证洗牌空窗 |
| 稳定性 | 最小战斗循环可以连续跑 3 个房间不阻塞 |

## 版本锁定规则

2026/05/11 进入内容锁定后：

- 不新增新卡牌类型。
- 不新增新的连招分支。
- 不新增新的关卡结构。
- 不改动冻结接口语义。
- 只修复 P0 / P1 问题。
- P2 问题记录到 512 后续迭代。

## 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 战斗侧和卡组侧接口理解不一致 | 联调失败 | 以 `协作接口/战斗与卡组接口契约.md` 为准 |
| 卡组 UI 信息太多 | 玩家看不懂 | 第一版只显示当前序列、洗牌、连携原因 |
| 洗牌空窗太惩罚 | 战斗节奏断裂 | 默认 1 秒，可通过武器 DA 调整 |
| 奖励进入卡组后顺序争议 | 构筑难以预测 | 第一版默认加入 DeckList 末尾 |
| 热度系统再次变复杂 | 抢占卡组心智 | 热度只做强化，不做激活门槛 |
