# AI 开发文档编写与分类规范

> 适用范围：使用 AI 进行需求分析、代码开发、配置交付、联调验收时产生的文档。  
> 适用人群：策划、程序、技术美术、使用 AI 协作开发的同事。  
> 核心原则：代码解决功能，文档解决交接；涉及配置的功能必须有策划可读的配置说明。

## 1. 什么时候必须写文档

| 场景 | 是否必须写 | 文档类型 |
| --- | --- | --- |
| 新功能开发完成，策划需要在引擎里配置 | 必须 | 配置说明 |
| 新增 DataAsset / DataTable / WBP / 材质 / Niagara / GameplayTag 配置 | 必须 | 对应配置说明 |
| 一个功能横跨多个系统，后续还会继续迭代 | 必须 | 开发方案 |
| 战斗、UI、关卡、奖励等系统有新规则 | 必须 | 系统设计或规则说明 |
| 只是修复小 bug，不涉及配置 | 可选 | 工作记录 |
| 临时讨论、过程推演 | 可选 | WorkSession 记录 |

## 2. 文档放置规则

AI 开发相关文档优先放在当前版本计划目录：

```text
Docs/项目需求文档/512版本项目计划/
```

### 2.1 开发方案

放在：

```text
Docs/项目需求文档/512版本项目计划/Codex开发方案/01_系统开发方案/
```

适用于 AI 要执行的开发计划、模块拆分。总览、上下文、索引和规范放在 `Codex开发方案/00_总览与规范/`。

命名建议：

```text
功能名开发方案.md
功能名后续开发方案.md
当前上下文.md
Codex开发总览.md
```

示例：

```text
01_系统开发方案/背包符文卡组开发方案.md
01_系统开发方案/符文连携系统后续开发方案.md
01_系统开发方案/战斗系统开发方案.md
```

### 2.2 策划配置说明

放在：

```text
Docs/项目需求文档/512版本项目计划/Codex开发方案/配置说明/
```

配置说明必须先按功能种类建文件夹，再按配置对象拆文件。

```text
配置说明/
  符文/
    战斗卡牌符文DA配置说明.md
    符文WBP配置说明.md
  武器/
    武器初始卡组配置说明.md
  UI/
    卡组HUD_WBP配置说明.md
  奖励/
    三选一奖励入卡组配置说明.md
  战斗手感/
    FreezeHitStop配置说明.md
```

拆分原则：

- 同一个功能如果涉及 DA 和 WBP，拆成两个文件。
- 同一个功能如果涉及材质和 Niagara，拆成两个文件。
- 策划只需要配置其中一个对象时，不应该被迫阅读另一个对象的配置。

### 2.3 设计文档

放在：

```text
Docs/项目需求文档/512版本项目计划/设计/
```

适用于玩法规则、卡牌规则、数值设计、字段规范等非代码文档。

推荐结构：

```text
设计/
  符文/
    卡牌设计总纲.md
    卡牌字段规范.md
    卡牌设计填写表.md
  战斗/
  关卡/
  UI/
```

### 2.4 系统技术文档

如果是长期系统说明，放在已有系统目录：

```text
Docs/Systems/
Docs/Design/
Docs/Survey/
```

示例：

```text
Docs/Systems/Rune/BuffFlow_ArchitectureAndPlan.md
Docs/Design/VFX_PoisonAndArmorShield_Niagara.md
```

### 2.5 工作过程记录

放在：

```text
Docs/WorkSession/
```

适用于阶段总结、代码审查记录、临时方案评审。

## 3. 配置说明文档写法

配置说明面向策划，目标是“照着文档能配出来”。

推荐模板：

```markdown
# 功能名配置说明

## 作用
一句话说明这个配置控制什么功能。

## 配置位置
写清楚在引擎哪里打开、哪个资产、哪个字段。

## 字段说明
| 字段 | 怎么填 | 影响 |
| --- | --- | --- |

## 推荐配置示例
给一套可以直接照抄的配置。

## 注意事项
写容易配错的地方。

## 测试方式
写策划在 PIE 或关卡里怎么确认生效。
```

写作要求：

- 少写代码实现，多写配置入口。
- 字段说明优先用表格。
- 每个配置对象单独成文。
- WBP 文档必须写清楚父类、组件层级、组件命名、是否需要勾选 `Is Variable`。
- 材质 Custom Node 文档必须说明 `.ush` 文件路径、Custom Node 输入、输出类型、材质节点连接方式。
- Niagara 文档必须说明 System / Emitter / User Parameter / Blueprint 传参位置。

## 4. 开发方案文档写法

开发方案面向程序和 AI，目标是“知道要改哪里、怎么验收”。

推荐模板：

```markdown
# 功能名开发方案

## Summary
本轮目标和不做什么。

## Key Changes
按系统列出要改的内容。

## Public APIs / Types
列新增类、函数、结构体、事件。

## Implementation Changes
写代码接入点和数据流。

## Test Plan
写自动化测试、编译验证、手动验收。

## Assumptions
写本轮默认假设，避免后续误解。
```

## 5. WBP 配置文档要求

只要功能涉及 WBP，就必须写清楚以下内容：

| 内容 | 必须写到什么程度 |
| --- | --- |
| 建议 WBP 名称 | 例如 `WBP_CombatDeckBar` |
| Parent Class | 例如 `CombatDeckBarWidget` |
| Designer 层级 | 用代码块列完整层级 |
| 组件变量名 | 写每个组件必须叫什么 |
| `Is Variable` | 写哪些组件必须勾选 |
| Graph 逻辑 | 写需要或不需要；如果不需要，明确写“Graph 可为空” |
| 推荐摆放 | 写锚点、尺寸、对齐方式 |
| 测试方式 | 写 PIE 中如何确认绑定成功 |

示例：

```text
Root: CanvasPanel
  CombatDeckBar: WBP_CombatDeckBar，勾选 Is Variable
```

## 6. 材质和 `.ush` 配置文档要求

只要功能涉及材质 Custom Node，必须同时提供：

```text
配置说明/功能种类/功能名材质配置说明.md
Source 或 Shaders 里的 .ush 文件
```

文档必须包含：

- `.ush` 文件路径。
- Custom Node 名称。
- 每个 Input 的名称、类型、默认值。
- Output 类型。
- 材质节点连接方式。
- 材质实例中需要开放给策划调的参数。
- 推荐参数示例。
- 常见错误，例如输入名不一致、shader 未重新编译。

## 7. 命名规范

| 类型 | 命名格式 | 示例 |
| --- | --- | --- |
| 开发方案 | `功能名开发方案.md` | `背包符文卡组开发方案.md` |
| 后续方案 | `功能名后续开发方案.md` | `符文连携系统后续开发方案.md` |
| 配置说明 | `配置对象配置说明.md` | `武器初始卡组配置说明.md` |
| WBP 配置 | `功能名_WBP配置说明.md` | `卡组HUD_WBP配置说明.md` |
| DA 配置 | `功能名DA配置说明.md` | `战斗卡牌符文DA配置说明.md` |
| 测试说明 | `功能名测试说明.md` | `卡组闭环测试说明.md` |

命名要求：

- 文件名要让策划能从名字判断用途。
- 不使用 `新建文本文档.md`、`说明.md`、`临时.md` 这类名称。
- 同一类配置放在同一文件夹下，不跨目录散放。

## 8. AI 开发交付清单

每次 AI 完成功能后，最终回复需要覆盖：

```text
1、完成了 XX 需求
2、制作了 XX 功能
3、需要 XX 配置，配置方案如下
4、需要测试 XX 方面
```

如果涉及配置，还必须检查是否新增或更新配置说明文档。

交付前自查：

- 是否有配置说明文档。
- 配置说明是否放在正确功能种类文件夹。
- WBP / DA / 材质 / Niagara 是否分文件写。
- 是否写了测试方式。
- 是否说明本轮不做什么。

## 9. 新需求开始前要问的问题

如果用户没有指定文档位置，AI 需要先问：

```text
这次配置说明文档放在哪个功能种类文件夹？
```

如果用户不确定，AI 可以按功能归类：

| 功能 | 默认文件夹 |
| --- | --- |
| 符文、卡牌、BuffFlow | `配置说明/符文/` |
| 武器、初始卡组、武器参数 | `配置说明/武器/` |
| HUD、菜单、WBP | `配置说明/UI/` |
| 掉落、三选一、奖励池 | `配置说明/奖励/` |
| HitStop、震屏、受击反馈 | `配置说明/战斗手感/` |
| Niagara、材质、后处理 | `配置说明/表现/` |

## 10. 最小文档标准

文档可以短，但不能缺以下信息：

```text
这个配置影响什么
在哪里配置
字段怎么填
推荐示例
怎么测试
```

如果一份文档无法回答这五点，就不能视为可交付配置说明。
