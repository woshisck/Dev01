# Docs 分类规则

> 更新：2026-05-15  
> 适用范围：`Docs/` 下所有手写文档。  
> 核心原则：按用途和功能分类，不按作者分类；长期知识和版本过程分开；不用的方案归档，不留在主入口。

## 一级分类

| 目录 | 使用场景 | 例子 |
| --- | --- | --- |
| `00_入口与规范/` | 文档入口、写作规范、迁移记录 | `分类规则.md`、`路径迁移表.md` |
| `01_长期系统文档/` | 跨版本长期有效的系统、代码、Tag、编码约定 | BuffFlow、战斗、UI、GameplayTag、Widget 规范 |
| `02_版本计划与需求/` | 核心方案、需求拆解、开发方案、协作接口 | 核心方案、接口契约、联调验收 |
| `03_配置与编辑器手册/` | 策划或 TA 在编辑器里配置的操作手册 | WBP、DA、材质、Niagara、DataEditor、Rune Editor |
| `04_调研与玩法设计/` | 调研报告、玩法规则、卡牌/符文设计 | Survey、卡牌字段规范、玩法方向 |
| `05_项目管理与进度/` | 任务、进度、验收、功能完成记录 | `TASKS.md`、`FeatureLog.md`、验收清单 |
| `90_自动生成报告/` | Commandlet、脚本、工具自动生成的报告 | SetupReport |
| `99_归档/` | 历史过程记录、旧 TODO、旧方案 | WorkSession、旧 2D 背包方案、临时记录 |

## 放置决策

| 我要写的内容 | 默认位置 |
| --- | --- |
| 系统架构、接口、数据流 | `01_长期系统文档/系统/<模块>/` |
| 编码规范、DataAsset/GAS/Widget/Material 约定 | `01_长期系统文档/编码规范/` |
| GameplayTag 或状态冲突规则 | `01_长期系统文档/标签/` |
| AI 或程序要执行的核心开发方案 | `02_版本计划与需求/核心方案/01_开发方案/<功能>/` |
| 核心需求、接口契约、联调验收 | `02_版本计划与需求/核心方案/` 下对应功能目录 |
| 策划配置说明 | `03_配置与编辑器手册/核心配置说明/<功能种类>/` |
| 编辑器工具使用说明 | `03_配置与编辑器手册/编辑器/` |
| 玩法规则、字段规范、卡牌设计表 | `04_调研与玩法设计/核心玩法设计/` |
| 调研和系统盘点 | `04_调研与玩法设计/调研总览/` |
| 当前任务和验收管理 | `05_项目管理与进度/` |
| 临时审查、过程计划、旧 TODO、旧方案 | `99_归档/` |

## 核心方案内部规则

`02_版本计划与需求/核心方案/` 按功能分类：

| 目录 | 内容 |
| --- | --- |
| `00_总览与规范/` | 核心方案总览、当前上下文 |
| `01_开发方案/<功能>/` | 各功能的执行方案和后续方案 |
| `02_联调验收/` | 联调、验收、跨系统测试方案 |
| `03_协作接口/` | 模块间接口契约 |
| `04_系统需求/<功能>/` | 更细的需求拆分、运行规则、数据结构 |

## 配置说明最低标准

配置说明可以短，但必须回答五个问题：

1. 这个配置影响什么。
2. 在编辑器哪里配置。
3. 字段怎么填。
4. 推荐示例是什么。
5. 怎么在 PIE 或关卡中测试。

如果涉及 WBP、DA、DataTable、材质 Custom Node、Niagara，应按对象拆文档，不把多种配置混在一篇里。

## 命名规则

| 类型 | 推荐命名 |
| --- | --- |
| 使用/配置指南 | `功能名_Guide.md` 或 `功能名配置说明.md` |
| 技术文档 | `功能名_Technical.md` |
| 设计说明 | `功能名_Design.md` 或中文规则名 |
| 参考手册 | `功能名_Reference.md` |
| 工作记录 | `功能名_WorkReport_YYYYMMDD.md` |

核心方案已经从 512 版本提升为主线，目录和文件名不再使用 `512` 作为分类前缀。资产名、历史记录、引擎路径中真实存在的 `512` 不强制改名。

## 归档规则

以下内容放入 `99_归档/`：

- 已被主文档吸收的旧 TODO。
- 临时过程记录、计划评审、代码审查记录。
- 已经不作为当前方向的旧方案，例如旧 2D 背包方案。
- 只用于追溯历史、不应作为最新配置依据的文档。

归档不等于删除。归档文档可以被引用，但引用处必须明确它是历史背景或旧记录。
