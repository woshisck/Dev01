# UE5.8 材质性能分级接口与制作技术规范

本文定义材质侧当前需要落地的接口和制作约束。当前阶段不直接完成具体材质网络，只提供性能分级接口、命名约定、贴图采样合同和打包 bake 入口，后续地面、墙面、多层混合、贴花混合和特殊效果材质可以按本文单独制作。

## 当前边界

- 只保留正式画质档位：`Epic` / `High` / `Mid` / `Low`。
- 美术制作 Source 材质时可以保持可视化效果完整，包含 mesh paint、顶点色、高度混合、静态贴花混合等。
- 打包或批处理阶段负责把静态结果 bake 成 VT Atlas 或 baked texture，不要求美术手动合并 atlas。
- 环境静态合批的贴图后端按 SVT/UDIM-style 虚拟大贴图方向推进：工具把普通 unique 贴图整理成 UDIM tile 组或等价 UDIM virtual texture，而不是生成必须整张常驻的传统超大 atlas。
- 角色、交互物、VFX、动态 gameplay actor 暂不进入环境合批，也不强制开启 VT。
- 本阶段不直接制作最终地面/墙面复杂母材，只固定接口和模板；材质网络可以后续由材质专项任务或美术侧实现。

## 当前已提供的材质模板

当前材质方案分为两条合同，不混在一张母材里：

| 合同 | 资产/入口 | 实现方式 | 说明 |
| --- | --- | --- | --- |
| Source 可编辑母材 | `M_Env_MasterA_Source`，已通过 UE MCP `MaterialTools` + `ProgrammaticToolset` 创建；也可由 `MaterialPerformanceTemplateSetup -Apply -Force` 重建 | 引擎默认材质节点，不依赖 `.ush` | 给美术和材质专项提供可见节点模板，包含 A/B/C BaseColor、高度混合输入、ORM、Normal、LightInfo、`Quality Switch` 和 `T_BatchVT_*` 合批贴图槽 |
| 材质四档分支 | UE 原生 `Material Quality Level` / `Quality Switch` | 由 `r.MaterialQualityLevel` 驱动 | 不要求用四套 MI 表达四档；MI 只作为普通资产变体或调参资产，不承担性能档位主逻辑 |
| Baked/VT 合批母材 | `M_Env_Baked_VTAtlas`，由 `MaterialBatchParentMaterialSetup -Apply -Force` 创建 | `TexCoord7.x -> _PropTexture -> VT_Atlas` 合同 | 给 GeneratedBatchProxy/Baked 使用，读取合批后的 baked/VT 结果 |

Source 模板不使用项目 `.ush`，原因是当前目标是给美术一个能在材质编辑器中直接查看和改造的节点合同。合批后的 `M_Env_Baked_VTAtlas` 仍可以包含 commandlet 生成的采样节点或 Custom 节点，因为它属于工具生成的运行时合同，不是美术日常编辑的复杂效果母材。

MCP 状态说明：当前 Codex 会话已经连接 UE MCP，并通过 `editor_toolset.toolsets.material.MaterialTools`、`editor_toolset.toolsets.material_instance.MaterialInstanceTools`、`editor_toolset.toolsets.programmatic.ProgrammaticToolset` 创建和反查了 Source 母材模板。`MaterialPerformanceTemplateSetup` 和 `MaterialBatchParentMaterialSetup` commandlet 仍保留为可重复生成和自动化 fallback。

不同性能效果处理位置：

- Source 材质里：通过 UE 原生 `Quality Switch` 决定 Low / Medium / High / Epic 的 shader 采样路径；游戏四档 `Low / Mid / High / Epic` 会显式映射到 UE 材质质量枚举。
- `Quality Switch` 约定：High/Epic 可保留更多 Source 混合路径；Medium 用简化混合；Low 优先使用 bake/合批后的轻量路径。具体采样数量由材质作者在 UE 节点中维护，不由四套 MI 强制表达。
- Baked/VT 路径里：通过 `M_Env_Baked_VTAtlas` 读取已经 bake/合并后的 VT Atlas 和 PropertyTexture，Mid/Low 不再依赖 Source 的多层运行时混合。
- 运行时设置里：通过 `FYogMaterialPerformanceTierInterface` 和 `r.Yog.MaterialQuality`、`r.Yog.DynamicOverlayQuality`、`r.Yog.MaterialLightQuality`、`r.Yog.MaterialLight.MaxLightInfoCount` 传递预算。

## 运行时接口

代码入口：

- `UYogPerformanceSettingsLibrary::GetMaterialPerformanceInterfaceForTargetTier`
- `UYogPerformanceSettingsLibrary::GetMaterialPerformanceInterfaceForGraphicsSettings`
- 输出结构体：`FYogMaterialPerformanceTierInterface`

材质和批处理系统应读取以下字段作为合同：

| 字段 | 用途 |
| --- | --- |
| `MaterialQuality` | 控制材质分支、混合层数、复杂函数开关 |
| `TextureQuality` | 控制贴图分辨率、streaming/mip 预算 |
| `VTAtlasQuality` | 控制生成 VT Atlas 的 cell 尺寸、mip 和通道质量 |
| `DynamicOverlayQuality` | 控制 runtime RVT/贴花/动态 overlay 的数量和质量 |
| `MaterialLightQuality` | 控制材质灯、LightInfo 或等价假光照预算 |
| `MaterialLightMaxLightInfoCount` | 材质灯最多可读 LightInfo 数量 |
| `MaxUniqueTextureSets` | Source 主材允许采样的 unique texture set 数量 |
| `MaxRuntimeBlendLayers` | runtime 材质允许保留的混合层数 |
| `MaxRuntimeOverlayLayers` | runtime overlay/RVT/动态贴花允许层数 |
| `bPreferSourceMasterMaterial` | 当前档位优先使用 Source 主材 |
| `bPreferBakedMaterial` | 当前档位优先使用 bake 后材质 |
| `bUseVTAtlas` | 环境合批材质默认走 VT Atlas |
| `bAllowGroundRuntimeRVTOverlay` | 地面 bake 后是否允许继续叠 runtime RVT/overlay |
| `bAllowWallRuntimeRVTOverlay` | 墙面是否默认允许 runtime RVT overlay；当前默认关闭 |
| `bBakeStaticDecalsIntoSurface` | 静态贴花是否进入 bake 结果 |
| `SourceMasterMaterialContract` | Source 主材合同名，默认 `M_Env_MasterA_Source` |
| `BakedMaterialContract` | bake 后材质合同名，默认 `M_Env_Baked_VTAtlas` |

## 四档材质策略

| 档位 | Source 主材 | Baked/VT Atlas | unique texture set | runtime 混合层 | runtime overlay | 典型目标 |
| --- | --- | --- | ---: | ---: | ---: | --- |
| Epic | 优先 | 可生成但不强制替换 | 3 | 3 | 2 | 高配 PC，保留完整视觉和对比路径 |
| High | 优先 | 选择性替换高压 cluster | 3 | 3 | 1 | 普通 PC，高质量 Source + 审核后的 proxy |
| Mid | 非优先 | 优先 | 2 | 2 | 1 | 默认推荐，主流 PC/掌机候选 |
| Low | 最少 | 强制优先 | 1 | 1 | 0 | 低配/省电，尽量使用 bake 后结果 |

解释：

- PC 侧主材 A 可以保留 2-3 套贴图采样和较完整的高度混合、顶点色混合、静态贴花混合。
- 掌机或低功耗路径不要求运行时重做完整混合，应优先使用当前效果 bake 后的贴图或 VT Atlas。
- Mid 可作为掌机候选验证档：允许 2 层 runtime 混合和 1 层 overlay，但最终是否保留由 profiling 决定。

## 贴图采样合同

Source 主材建议固定以下通道命名，方便 audit 和 bake 工具自动识别：

| 通道 | 建议参数名 | 内容 |
| --- | --- | --- |
| BaseColor | `T_BaseColor_A/B/C` | 颜色或 albedo |
| Normal | `T_Normal_A/B/C` | tangent normal |
| ORM | `T_ORM_A/B/C` | R=AO, G=Roughness, B=Metallic |
| Height | `T_Height_A/B/C` | 高度混合权重 |
| Mask | `T_Mask_A/B/C` | 特殊遮罩、湿度、破损等 |
| LightInfo | `T_LightInfo` | 材质灯或静态假光照数据，独立于普通 atlas |

可接受后缀表由 `UYogMaterialTextureNamingConvention` 资产维护，默认资产为 `/Game/Generated/Performance/DA_MaterialTextureNamingRules_Default`：

| 通道 | 推荐后缀 | 兼容后缀 | 说明 |
| --- | --- | --- | --- |
| BaseColor | `_BaseColor`, `_BC` | `_D`, `_Diffuse`, `_Albedo`, `_A` | `_A` 仅用于兼容 Albedo 命名，和 A/B/C set 后缀同时出现时优先按参数合同判断 |
| Normal | `_Normal`, `_N` | `_NM`, `_NRM` | 必须使用 normal map 压缩和线性采样 |
| ORM | `_ORM` | `_MRA`, `_RMA`, `_Packed`, `_MaskMap` | 默认 R=AO、G=Roughness、B=Metallic |
| Height | `_Height`, `_H` | `_Disp`, `_Displacement` | Source/bake 输入，不要求 Low runtime 采样 |
| Mask | `_Mask`, `_MSK` | `_Masks`, `_BlendMask`, `_UniqueMask` | 项目自定义遮罩，合批前必须明确通道含义 |
| LightInfo | `_LightInfo` | `_LI`, `_FakeLight`, `_MatLight` | 独立材质灯/假光照输入，不进入 ORM |
| Emissive | `_Emissive`, `_E` | `_Emission`, `_Glow` | 可选通道，不要求所有环境材质都有 |

约束：

- 同一个材质族内，贴图 set 后缀必须保持一致：`A/B/C`。
- `Low` 不应依赖 B/C set 的运行时采样；需要先 bake 到单套结果。
- `Height` 只作为 Source 混合输入或 bake 输入，不要求在 Low runtime 保留。
- `LightInfo` 不进入普通 ORM；它由 `MaterialLightQuality` 和 `MaterialLightMaxLightInfoCount` 单独控制。

## Batch 材质采样合同

环境 batch proxy 使用以下合同：

- `BatchMaterialIndex` 写入 `TexCoord7.x`。
- Property Texture 以 `BatchMaterialIndex` 为行索引。
- 每行至少包含 BaseColor、Normal、ORM、Emissive、Mask 的 slice 或 VT rect 信息。
- VT Atlas 模式下，每个通道记录独立 UV rect：BaseColor、Normal、ORM 可有不同 rect。
- 材质父级只负责根据 `BatchMaterialIndex` 读取属性，不负责重新推导美术混合逻辑。

## SVT/UDIM 虚拟大贴图合批合同

结论：可以采用 UDIM/SVT 方式实现“虚拟大贴图合批管线”。它比传统超大 atlas 更适合本项目，因为运行时按 virtual texture page/tile 需求驻留，不要求把逻辑大图整张加载进普通 streaming pool。

边界：
- UE 支持 UDIM 命名导入和 UDIM virtual texture 资产，但不会自动把项目里已有的普通单体贴图、材质、mesh UV 整理成合批 UDIM。
- 该能力不等于 draw call 合批；draw call 仍要靠 Merge Mesh / section 合并 / Source-Proxy-Baked 切换。
- UDIM/SVT 仍有 page table、physical pool、tile border、feedback 和 shader virtual sampler 成本，必须进入 `ResidencyRiskPlan` 和 profiling。
- Batch 父材质的贴图参数必须使用 Virtual sampler 类型；普通 Texture Sample 参数不能直接混接 virtual texture。

目标工具链：
```text
收集静态环境模型和 Source 材质贴图
-> 按合批 cluster 分配 UDIM tile
-> 重排合批 mesh 的材质 UV 到对应 UDIM tile
-> 生成 BaseColor / Normal / ORM 等 UDIM 贴图组
-> 导入或创建 UE UDIM Virtual Texture
-> 生成统一 batch 材质实例，绑定 UDIM VT + PropertyTexture
-> Merge Mesh，并压成一个材质 section
-> 写入 MappingData、ResidencyRiskPlan、Source/Proxy/Baked readiness
```

美术侧要求：
- 美术继续按单体资产制作普通材质和 unique 贴图，不手工维护 UDIM tile。
- 工具负责 tile 排布、UV 重排、padding、mip bleed 保护、命名、导入、材质实例绑定和报告。
- Source 材质命名合同仍保留，保证工具能识别 BaseColor、Normal、ORM、Height、Mask、LightInfo。
- 如果某个资产需要退出合批，使用 `EnvBatch.Exclude`，不要求美术维护另一套 UDIM。

生成命名建议：
| 类型 | 命名 |
| --- | --- |
| UDIM BaseColor tile | `T_UDIM_<Cluster>_BC.1001.png` |
| UDIM Normal tile | `T_UDIM_<Cluster>_N.1001.png` |
| UDIM ORM tile | `T_UDIM_<Cluster>_ORM.1001.png` |
| UE UDIM VT asset | `VT_UDIM_<Cluster>_<Channel>` |
| Batch MI | `MI_Env_BatchUDIM_<Cluster>` |
| Mapping Data | `DA_MaterialBatchMap_<Cluster>` |

验收补充：
- dry-run report 必须列出 UDIM tile 分配、每个 source texture 对应 tile、UV 重排状态和 tile padding。
- 真实验收必须记录 `stat virtualtexturing`、`stat virtualtexturememory`、普通 texture streaming pool、batch 前后 draw call / mesh draw command。
- 如果 Source 普通贴图和 UDIM VT 合批结果同时常驻，必须标记 duplicate residency risk，并要求 Source/Proxy/Baked 加载互斥后再进入提交。

| 类型 | 命名 |
| --- | --- |

| Batch MI | `MI_Env_Batch_<Cluster>` |
| VT Atlas | `VT_Atlas_<Cluster>` |
| Property Texture | `T_PropTexture_<Cluster>` |
| Mapping Data | `DA_MaterialBatchMap_<Cluster>` |
| Baked BaseColor | `T_Baked_<Cluster>_BC` |

| Baked ORM | `T_Baked_<Cluster>_ORM` |

## 地面与墙面

地面：

- Source 阶段允许多层材质、高度混合、mesh paint/顶点色和静态贴花混合。
- bake 阶段把静态结果写入 VT Atlas 或 baked texture。
- bake 后的 runtime 动态效果只在 RVT/overlay 层继续叠加，例如脚印、湿痕、临时污染。
- `Low` 默认关闭非必要动态 overlay。

墙面：

- Source 阶段同样允许 mesh paint/顶点色、高度混合和静态贴花混合。
- 默认不推荐墙面 runtime RVT overlay；优先做静态 bake 到 wall baked texture 或 VT Atlas。
- 如墙面确实需要动态变化，应作为独立 runtime overlay 材质或局部 decals 处理，不进入静态 batch bake。
- 墙面和地面可以共用一套接口，但建议保留不同母材合同：`M_Env_Ground_Source`、`M_Env_Wall_Source`、`M_Env_Baked_VTAtlas`。是否拆不同母材由具体材质复杂度决定，不应影响 batch index 和 bake 输出合同。

## VT 与非 VT 混用

- 环境 batch 默认优先 VT Atlas。
- 角色、交互物、VFX、动态物暂不强制 VT，也不进入合批。
- 需要持续记录 VT/non-VT residency 风险：混用可能启用两套 texture residency/pool 预算，带来显存或内存上升。
- 验收时必须同时记录：
  - VT pool 使用量；
  - non-VT streaming pool 使用量；
  - batch 前后材质采样数；
  - batch 前后 draw call / mesh draw command；
  - 是否出现重复保留 Source texture 与 baked/atlas texture 的情况。

## 美术制作要求

- 美术提供单独 unique 贴图，不需要手动合 atlas。
- 贴图和材质参数必须遵守命名合同，保证 commandlet 可以自动识别。
- Source 效果以所见即所得为目标；bake 结果由自动化报告验证差异。
- 静态贴花若需要进入最终表面，标记为 static bake decal；动态贴花必须保持 runtime decal，不进入 baked 贴图。
- 任何不确定是否可合批的对象先加 `EnvBatch.Exclude`，再由 audit 报告移除。

## 验收口径

材质接口阶段验收：

- C++ 编译通过。
- `DevKit.Performance.Settings.MaterialTierInterface` 自动化测试通过或至少可被 AutomationTest 发现。
- 文档中四档策略、采样合同、命名合同、地面/墙面处理、VT/non-VT 风险记录完整。
- MaterialBatch dry-run 报告能记录 Source/Proxy、VT Atlas、Property Texture、residency 风险和实际 StreamingLevel layer 阻塞点。

材质制作阶段验收：

- Source 与 baked 视口对比截图。
- bake 前后材质采样数对比。
- VT/non-VT pool 对比。
- Epic/High/Mid/Low 四档下 runtime CVar 和材质接口输出一致。
- Low/Mid 不依赖 Source 多层混合才能保持基本视觉。
