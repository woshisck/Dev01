# UE5.8 合批与材质烘培接口状态及后续计划

日期：2026-07-01  
项目：DevKit / UE5.8  
当前目标平台优先级：PC 端优先

## 1. 当前结论

当前项目已经具备“静态环境合批”和“材质烘培/VT Atlas”方向的基础接口，但还没有完成正式生产闭环。

已经具备的部分：

- `EnvBatch.*` Actor Tag 标记入口。
- `MaterialBatchBuild` commandlet 的扫描、dry-run、report、manifest、分阶段生成接口。
- Mapping DataAsset、PropertyTexture、TextureArray、VT Atlas、ProxyMesh、BatchMaterial 的生成阶段接口。
- 模型资产合规检查编辑器，用于检查 StaticMesh 的 LOD、材质槽、碰撞、分类和排除状态。
- 材质合规检查编辑器，用于检查贴图命名、sRGB、尺寸、VT 建议，以及材质 Source/Baked/VT Atlas 接口。
- UE 原生 `r.MaterialQualityLevel` 已接入项目四档性能设置，材质 `Quality Switch` 可以按 UE 原生逻辑工作。

尚未完成的部分：

- 正式打包前自动替换关卡资产的完整执行链。
- 墙面/建筑复杂材质的“编辑器所见即所得 -> 出包前烘培成贴图/VT”的完整视觉生产链。
- 真实关卡 cluster 的 VT Atlas 生成、视觉对比、VT physical pool / streaming pool profiling。
- 最终合批父材质的完整采样规范、视觉验收和跨场景压力测试。

因此当前状态应定义为：工具链和数据契约已具备第一版，正式生产替换和材质烘培闭环仍是后续任务。

## 2. 当前技术方向

### 2.1 PC 端优先

当前主要目标是 PC 端，因此模型性能处理先收敛到 LOD 合规：

- Epic / High：默认使用 LOD0。
- Mid / Low：允许使用 LOD1 或后续代理版本。
- 当前阶段不强制做复杂的运行时模型四档替换。
- 合批属于打包/生产链优化，不作为日常美术编辑时的运行时切换逻辑。

### 2.2 材质分为三块处理

#### 地面

地面使用一套 RVT 主材。

地面不通过 Epic / High / Mid / Low 做材质分支。地面的多层混合、贴花混合、Mesh Paint、顶点色等静态效果应在 RVT 写入前完成。动态效果如果需要，应放在 RVT 之后的位置继续叠加处理。

当前结论：可行，作为地面主线保留。

#### 墙面、建筑

墙面和建筑可能需要：

- 2 层材质混合。
- 顶点色或 Mesh Paint 控制。
- 可能接收 RVT 或类似的环境效果。
- 静态贴花混合。
- 后续出包前烘培成默认贴图或 VT 结果。

这部分暂时搁置，不作为当前必须完成的性能分级功能。后续建议单独做墙面/建筑材质生产链：编辑器阶段保留可编辑混合效果，打包阶段生成 baked 版本，再由替换链接入合批或简化材质。

当前结论：方向可行，但不应在本阶段强行接入完整烘培闭环。

#### 静态模型、物件、非交互道具

静态环境物件继续走合批逻辑：

- 美术继续制作普通单体资产。
- 美术继续提交 unique 贴图，不手工维护 UDIM、Atlas 或 SVT tile。
- 关卡中通过 Actor Tag 标记需要进入合批的对象。
- 正式打包链根据 `EnvBatch.Source.<Group>` 收集对象。
- 工具自动生成 VT/UDIM-style Atlas、Mapping、PropertyTexture、ProxyMesh、BatchMaterial 和替换计划。
- 交互物、角色、武器、运行时动态物件默认排除，不进入静态环境合批。

当前结论：这是当前最适合继续推进的主线。

## 3. 性能分级调用方式

项目四档仍面向玩家和美术显示为：

- Epic
- High
- Mid
- Low

运行时材质分级主线使用 UE 原生接口：

- `r.MaterialQualityLevel`
- 材质节点中的 `Quality Switch`
- UE Scalability / DeviceProfile

项目自定义 CVar 只作为增强预算和生产链参数：

- `r.Yog.MaterialQuality`
- `r.Yog.DynamicOverlayQuality`
- `r.Yog.BatchProxyPreference`
- `r.Yog.TextureCollectionQuality`
- `r.Yog.MaterialLightQuality`
- `r.Yog.MaterialLight.MaxLightInfoCount`

注意 UE 材质质量枚举顺序是：

- Low = 0
- High = 1
- Medium = 2
- Epic = 3

项目自己的 UI 顺序是：

- Low = 0
- Mid = 1
- High = 2
- Epic = 3

运行时会把项目四档显式映射到 UE 原生材质质量档位。

## 4. 已完成工作进展

### 4.1 模型侧

- 已完成模型资产合规检查编辑器第一版。
- 支持 `/Game/Art` 下 StaticMesh 扫描。
- 支持基于文件夹的目录树预览。
- 支持模型分类筛选。
- 支持状态设置：自动、通过、警告、阻断、排除。
- 右侧修改状态后，左侧目录展开位置和选择状态会尽量保留。
- 支持打开模型资产和在 Content Browser 中定位模型。
- 已取消独立模型分级配置资产路线，模型 LOD 信息回归 StaticMesh 自身维护。

### 4.2 场景合批侧

- `环境合批标记` 只负责打 Actor Tag。
- 支持 `EnvBatch.Source.<Group>`、`EnvBatch.Proxy.*`、`EnvBatch.Baked.*`、`EnvBatch.Exclude` 等语义。
- 合批组可以是不连续 Actor。
- 普通 StaticMeshActor、带 StaticMeshComponent 的蓝图、处理好的 Level Instance 边界对象都可以作为输入。
- 正式合批执行不暴露为美术日常按钮，保留在打包/TA 自动化链执行。

### 4.3 材质与贴图侧

- 已把原“贴图命名规则”入口调整为“材质合规检查”。
- 支持写入默认贴图命名规则资产。
- 支持检查 Texture2D 的后缀、sRGB、尺寸和 VT 建议。
- 支持检查 Material / MaterialInstance 是否暴露 Source/Baked/VT Atlas 所需参数。
- 保留 UE 原生 `Quality Switch` 作为材质性能分级主线。
- 不再依赖额外 MI 套娃方案作为性能分级主线。

### 4.4 合批与烘培接口侧

- `MaterialBatchBuild` 支持 dry-run 报告和 manifest。
- 已具备 Mapping DataAsset、PropertyTexture、VT Atlas、ProxyMesh、BatchMaterial 等分阶段接口。
- 全量 `-Apply` 替换仍应保持谨慎，不作为当前美术日常入口。
- `-ApplyVTAtlasOnly` 等 partial apply 适合后续由 TA 或打包链审核后执行。

## 5. 后续任务计划

### P0：保持当前工具可用

- 编译通过后提交并推送 main。
- 美术先使用模型资产合规检查修正 StaticMesh LOD0/LOD1。
- 美术使用材质合规检查修正贴图命名、sRGB、尺寸和基础材质接口。
- 场景侧只使用 `环境合批标记` 打 tag，不直接执行合批替换。

### P1：真实关卡 dry-run 验证

- 选一个真实 Streaming Level 作为样例 cluster。
- 给静态环境物件标记 `EnvBatch.Source.<Group>`。
- 运行 `MaterialBatchBuild` dry-run。
- 审查候选、拒绝原因、材质槽、贴图通道、VT Atlas 计划和 ProxyMesh 计划。
- 输出可复查的 report / manifest。

### P2：VT Atlas / UDIM-style 生产链验证

- 使用真实 unique 贴图验证自动收集和通道识别。
- 验证 VT Atlas 生成尺寸、tile、mip、格式和内存估算。
- 验证 VT 与非 VT 混用风险，重点观察 VT physical pool 和普通 streaming pool 是否出现重复驻留。
- 确认 PC 端默认规格，暂不为掌机端过早降级。

### P3：合批代理和替换链验证

- 生成 ProxyMesh。
- 验证 `TexCoord7.x` 或等价 batch index 写入。
- 验证 PropertyTexture 行索引和材质采样是否一致。
- 验证合并后材质槽数量、draw call、三角面、碰撞和遮挡表现。
- 在真实关卡中对比源版本和代理版本视觉差异。

### P4：墙面/建筑材质烘培专项

- 单独定义墙面/建筑主材规范。
- 明确 2 层混合、顶点色、Mesh Paint、贴花、RVT 接收或替代方案。
- 设计编辑器可编辑材质和出包 baked 材质之间的转换规则。
- 验证烘培后的贴图是否能进入 VT Atlas 或独立 baked texture。
- 这部分不强制纳入当前合批主线，避免影响现有模型合规和场景 tag 流程。

### P5：打包链 gate

- 把模型合规、材质合规、EnvBatch tag 审查、MaterialBatchBuild dry-run 纳入打包前检查。
- 对阻断项直接失败，对警告项输出报告。
- 建立一套 PC 端验收基线：帧率、draw call、GPU time、streaming pool、VT pool、视觉差异截图。

## 6. 注意事项

- 地面 RVT 不需要强行做性能分级分支。
- 墙面/建筑材质烘培暂时不要混入当前模型合规工具。
- 交互物、角色、武器和运行时动态物件默认不进入静态环境合批。
- 美术不需要手工维护 UDIM、SVT tile 或 Atlas。
- `Quality Switch` 是 UE 原生材质分级接口，可以用于需要分支的材质，但不是所有材质都必须使用。
- 项目 CVar 用于增强预算和生产链控制，不替代 UE 原生 Scalability。
- 当前合批/烘培接口已经存在，但正式生产替换仍需要真实关卡验证后才能开放。
