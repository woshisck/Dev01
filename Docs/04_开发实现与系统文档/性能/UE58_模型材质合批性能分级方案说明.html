UE5.8 模型与材质合批性能分级方案说明

目标是解释后续我会如何处理性能分级，尤其是模型合批、材质合批、自动化分组、AI/MCP 辅助、平台档位和验证标准。本文是方案说明，不代表已经执行合批替换。

总判断 平台分级 模型合批 材质合批 EnvLitBatch 对照 美术工作流 自动化流程 俯视角/RVT Material Cache 验证指标 实施阶段

1. 总判断

结论：模型和材质合批可以做，但不能设计成“美术摆完场景后工具无脑自动合并整个关卡”。正确方向是：美术表达分组意图，工具做规则审计，AI/MCP 做建议和批量操作，最终只对审查通过的静态环境 cluster 生成合批资产。

我会把整个系统拆成四层，不把它们混为一谈：

美术组织层

Actor Folder、Data Layer、Level Instance、选择集、或将来可能的 BP_EnvBatchGroup。这层表达“哪些物件属于一个区域”。

执行标记层

EnvBatch.Source.*、EnvBatch.Proxy.*、EnvBatch.Exclude。这是 commandlet 可以稳定读取的事实输入。

生成资产层

MappingDataAsset、Texture2DArray、PropertyTexture、ProxyMesh、BatchMaterial。

运行分级层

PC/掌机不同档位决定显示 Source、Proxy、HLOD、Baked 替代、Lumen Lite 或 Lumen Off。

这个拆法的好处是：美术不用理解 Texture2DArray 和 PropertyTexture，工具也不会因为文件夹拖错就直接破坏场景。

2. 性能分级会从哪些平台目标出发

目标档位 视觉策略 模型策略 材质策略 光照策略
PC Ultra 最高画质，优先保留原始资产细节。 Source 为主；只对重复远景或明显高 draw call 区域使用 HLOD/Proxy。 允许较完整材质功能，材质灯采样预算最高。 允许 Lumen High/Ultra；Nanite/VSM 当前不作为掌机依赖。
PC High 接近 Ultra，但降低部分阴影、反射和材质灯预算。 Source + 选择性 Proxy。 保持主要材质表现，减少非关键分支。 可开 Lumen，质量低于 Ultra。
Steam Deck 15W 掌机主验证档，目标是稳定视觉和 GPU ms。 房间/走廊 cluster 使用 ProxyMesh、HLOD、ISM/HISM。 优先批处理材质，减少材质实例导致的 draw call 拆分。 可测试 Lumen Lite，但是否默认开启必须看实测。
Switch 2 Candidate 候选档，不当作真机结论。 复用掌机 15W 策略，保守一点。 材质功能收敛，减少 unique 采样压力。 暂按 Lumen Lite 候选，不做最终承诺。
Steam Deck 5W / Low 低功耗档，优先稳定性和可读性。 Proxy / HLOD / Baked 替代优先，Source 尽量减少。 批材质 + 降采样 + 降材质灯预算。 Lumen Off，使用烘焙/静态补光/材质灯低档。
注意：性能分级不是只改一个画质选项。模型数量、材质 slot、材质实例、贴图 streaming、动态灯、阴影、Lumen、分辨率比例都会互相影响。

3. 模型合批会从哪些角度判断

3.1 先判断瓶颈类型

模型合批不是永远第一优先级。我会先判断场景压力来自哪里：

压力类型现象优先处理方式
Draw Call / Mesh Draw Commands 过高 大量小模型、材质实例多、section 多。 Geometry Merge、HLOD、ISM/HISM、批处理材质。
顶点/三角形过高 GPU ms 高，但 draw call 未必很夸张。 LOD、ProxyMesh 简化、HLOD、可见性分块。
材质像素成本高 材质采样多、材质灯采样多、透明/Masked 压力大。 材质分级、减少采样、预烘焙、批材质简化。
贴图显存/streaming 压力 纹理池爆、mip 抖动、加载 hitch。 VT/SVT、纹理尺寸规范、合并贴图集，但这不直接等于 draw call 合批。
动态灯/阴影压力 Shadow / Light pass 成本高。 减少动态灯、烘焙替代、阴影分级、材质灯预算分级。

3.2 哪些模型适合合批

静态环境 不交互 不破坏 不脚本移动 材质父级相近 同一区域可见性一致

适合第一批处理的是墙体、地面、柱子、固定建筑件、远景装饰、重复小件、地牢走廊模块这类资产。

3.3 哪些模型不应合批

门 机关 宝箱 拾取物 敌人 武器 可破坏物 剧情触发器 独立碰撞事件

这些对象即便视觉上像环境件，也应该默认 EnvBatch.Exclude，除非关卡/程序明确确认。

3.4 合批方式选择

方式适合情况优点风险
ISM/HISM 同 mesh、同材质、重复摆放。 风险低，适合重复道具。 不同 mesh/材质帮助有限。
Geometry Merge + Batch Material 同一区域多个静态 mesh，希望降低 draw call。 能把不同 mesh 合成代理，draw call 优化明显。 需要处理 UV、材质 index、贴图数组、碰撞和视觉差异。
HLOD / ProxyMesh 中远景、房间外、低配档。 成熟、适合分级切换。 近景可能视觉差异明显。
Packed Level Actor / Level Instance 模块化房间或 prefab-like 场景块。 美术组织友好。 本身不等于材质合批，需要结合 Proxy 和 batch material。

4. 材质合批会从哪些角度判断

4.1 材质合批的核心前提

材质合批不是“MI 父材质相同就一定能合批”。我会按下面条件审查：

  • 父材质或 shader path 是否一致。
  • Static Switch / keyword 组合是否一致。
  • Blend Mode、Shading Model、Two Sided、WPO、Masked 等渲染状态是否一致。
  • Texture 参数集合是否一致，例如 BaseColor、Normal、ORM、Mask。
  • 贴图尺寸、格式、mip、sRGB 是否可打入 Texture2DArray。
  • Scalar/Vector 参数能否进入 PropertyTexture。
  • 材质灯、LightInfo、特殊遮罩、全局 mask 是否需要独立降级路径。

4.2 M_Env_Building 的批材质方向

当前方向是做一个 M_Env_Building_Batch 或等价材质函数路径：

Geometry Merge 后的大 Mesh
  Vertex / UV7.x 存 MaterialIndex
  Fragment 阶段读取 MaterialIndex
  MaterialIndex -> Texture2DArray slice
  MaterialIndex -> PropertyTexture row
  输出 BaseColor / Normal / ORM / Mask / 参数化材质灯预算

其中 Tex_LightInfo 暂时不应混进普通 BaseColor/Normal/ORM 数组。它更像材质灯/CanvasRenderTarget 特殊数据，应该独立按质量档控制。

4.3 Texture2DArray 和 PropertyTexture 分工

数据进入哪里说明
BaseColor / Albedo T2DA_BaseColor 每个材质一层 slice，尺寸格式必须统一或预处理。
Normal T2DA_Normal 注意 normal 格式、压缩和切线空间一致。
ORM / MRA T2DA_ORM 通道约定必须固定，不能一个材质 R=AO，另一个 R=Metallic。
Mask / Unique Mask T2DA_Mask 或独立路径 根据使用频率决定是否纳入批材质。
颜色、强度、UV ST、开关运行时等价参数 PropertyTexture 每行对应一个 material index，每列对应一个参数。
Static Switch 不能直接运行时变化 应拆为少量 batch variant，或改造成运行时参数。
不会做的事：不会把所有 MI 不审查地塞进一个 Texture2DArray，也不会把 VT/SVT 当作 draw call 合批本身。VT 可以解决 streaming/显存，但 draw call 下降主要来自 Geometry Merge、HLOD、ISM/HISM 和批材质。

4.4 如果所有物件都使用虚拟纹理，能否替代 Texture2DArray 合批

结论：“所有贴图启用 Streaming Virtual Texture”不能自动实现材质合批；“工具链生成共享 VT Atlas / Material Cache，并让批材质按 materialIndex + UVRect 采样”才可以把 VT 作为 Texture2DArray 的替代资源层。换句话说，VT 可以替代 array 的贴图承载方式，但不能替代 batchIndex、PropertyTexture、材质变体收敛和 Mesh 合并。
做法是否减少 draw call原因是否推荐
每张 unique 贴图单独勾选 SVT 否 UE 仍然把它们当作不同纹理参数/材质资源绑定；只改善 streaming 和显存页加载，不会把多个 StaticMeshActor 合成一个 draw call。 可作为显存优化，但不是合批方案。
共享 VT Atlas + 批材质 可以 所有材质采样同一套虚拟纹理资源，shader 通过 materialIndex 查 UVRect 和参数表，类似 T2DA Batch。 可作为第一阶段主线，但必须由工具自动生成共享 VT Atlas 和 MappingData。
RVT 地面缓存 间接可能 减少地面 decal、混合层、复杂材质叠加成本，但不自动合并建筑/道具 draw call。 推荐用于俯视角地面管线。
UE5.8 Material Cache 不直接 缓存材质属性结果，偏向降低复杂材质求值成本；仍需要 Mesh 合并或实例化才能减少 draw call。 只作为实验验证项。
Texture2DArray + PropertyTexture 可以 采样逻辑确定、实现简单、与 EnvLitBatch 对齐；缺点是尺寸/格式/slice 数有约束。 推荐作为第一阶段确定性方案。

4.4.1 为什么“单独开启 SVT”不能直接合批

SVT 的底层物理页池可以复用显存管理，但材质层面仍然需要知道采样哪一张虚拟纹理、哪个 page table、哪些参数。只要每个 MI 仍然绑定不同纹理参数、不同 static switch、不同 render state，Mesh Draw Command 仍会被拆开。

真正的合批需要运行时只绑定一套批材质资源，然后每个像素根据 materialIndex 自己查表：

TexCoord7.x
  -> materialIndex
  -> PropertyTexture[materialIndex]
  -> TextureMapping[materialIndex] = VTRect / PageInfo / UVScaleOffset
  -> Shared Virtual Texture sample
  -> 批材质统一输出

4.4.2 如果用 VT 替代 Texture2DArray，需要补齐哪些工具链

  1. 生成共享虚拟纹理资源：不是每个材质一张独立 VT，而是按 batch group 生成统一的 VT Atlas / cache。
  2. 生成 TextureMapping：每个 materialIndex 对应 BaseColor、Normal、ORM 等通道在 VT 中的 rect、mip/scale、边界信息。
  3. 继续生成 PropertyTexture：颜色、UV ST、材质灯预算、scalar/vector 参数仍然需要按行查表。
  4. 继续写入 batchIndex：合并 Mesh 后顶点仍要知道自己属于哪个材质。
  5. 继续做材质兼容审计：static switch、blend mode、masked、two sided、WPO、材质灯分支不一致仍然会拆变体。
  6. 继续做 Geometry Merge / ISM / HISM / HLOD：VT 只解决“贴图资源怎么统一采样”，不解决“多个 Mesh 怎么变成少量 draw call”。

4.4.3 当前建议的两阶段路线

阶段贴图承载目标原因
Phase A Selective VT Atlas Batch 项目可以开启 VT/RVT 支持，但只对合批候选生成共享 VT Atlas；同时跑通 batchIndex、PropertyTexture、ProxyMesh、材质 parity。 兼容后续 RVT 方向，不要求所有原始贴图都转 VT，也避免 Texture2DArray 在 VT 管线下的限制。
Phase B Texture2DArray fallback 为不适合 VT Atlas 的平台或特殊批次保留 T2DA 生成路径。 T2DA 仍可作为确定性 fallback，但不作为唯一主线。
并行地面线 RVT 缓存俯视角地面混合、贴花、污渍、边缘融合。 地面表现需求和建筑/道具合批不同，应该单独管线处理。

4.4.4 Selective VT Atlas Batch 的自动处理方式

如果第一阶段就要依赖 VT，推荐做成“选择性 VT Atlas 合批”，不是把全项目所有贴图都转成 VT。工具只对进入 batch group 的材质生成共享 VT Atlas，原始材质实例和原始贴图仍然作为美术源数据保留。

这样做的关键是：美术不需要制作 Array，也不需要维护 VT Atlas。工具在合批资产目录下生成批处理专用虚拟纹理资源，批材质只绑定这些共享资源。

5. EnvLitBatch 机制对照：我们真正要复刻什么

新的关键结论：附件里的 EnvLitBatch 不是“开 VT 后自动合批”，而是一套完整的离线数据重排系统。它能合批，是因为离线阶段同时完成了 Mesh 合并、per-vertex batchIndex 写入、Texture2DArray 打包、UVRect/贴图索引表、PropertyTexture 属性表，并且运行时只用一套 batch shader 读取这些数据。

5.1 EnvLitBatch 的完整链路

1. 收集材质找到同 shader 变体、同 keyword 组合、可兼容的材质实例。
2. 打包贴图BaseColor、Normal、PackedMap 等进入 Texture2DArray 或等价 atlas。
3. 写属性表所有 scalar/vector 参数写入 _PropTexture，每行对应一个材质。
4. 写贴图索引每个材质的 slice index、UVRect 写入 _PropTextureArrayInfo 或 CBuffer。
5. 合并 Mesh多个 mesh 离线合成一个代理 mesh。
6. 写 BatchIndex每个顶点的 UV 通道写入该顶点所属材质 ID。

5.2 运行时 shader 如何知道“这个像素属于哪个材质”

EnvLitBatch 的核心不是 Texture2DArray 本身，而是 per-vertex batchIndex。离线合并 mesh 时，每个原始物体/材质对应一个材质编号，写进顶点 UV 的某个通道，例如 UV.a。Vertex/Fragment 阶段再读取这个值，得到当前像素应该使用哪一行属性、哪一个贴图 slice。

Vertex UV.a
  -> batchIndex
  -> _PropTexture[batchIndex, propertyColumn]
  -> _PropTextureArrayInfo[batchIndex]
  -> Texture2DArray[sliceIndex, remappedUV]
  -> 正常 PBR / EnvLit 计算
新增确认：batchIndex 应完全由工具链自动生成，美术不需要也不应该手动设置。Lilith EnvLitBatch 的逻辑是在 Mesh 合并时用循环序号 indexIter 分配 batchIndex，并写入合并后顶点 UV 数据；美术只正常摆放使用 EnvLit Shader 的 MeshRenderer / StaticMeshActor。

这意味着 UE 方案里也必须有一个明确的“材质 ID 来源”。它可以来自：

  • Geometry Merge 时写入某个 UV 通道，例如 TexCoord7.x。
  • 合并 mesh 的 vertex color / custom attribute。
  • ISM/HISM 场景下的 Per Instance Custom Data，适合实例级而不是三角面级材质差异。
  • ProxyMesh 生成时的 section/material remap 表。

需要注意平台差异：Unity 示例可以把原始 UV0 扩展成 float4，用 UV0.w 存 batchIndex；UE Static Mesh 的普通 UV 通道更适合按 float2 多通道处理，所以当前 UE 方案建议把 batchIndex 写入预留通道，例如 TexCoord7.x，而不是要求美术提供 UV0.w。

5.2.1 batchIndex 自动生成规则

UE 工具链应复刻“自动分配，不暴露给美术”的原则：

收集候选 StaticMeshComponent
  -> 按 batch group / 材质兼容性 / shader variant 分桶
  -> 为每个桶内的 material slot 分配稳定 materialIndex
  -> 合并 mesh section
  -> 把 materialIndex 写入合并后顶点 TexCoord7.x
  -> 生成 Texture2DArray / PropertyTexture / MappingData
  -> 生成或更新 BatchMaterialInstance
环节是否美术操作自动化规则
batchIndex 分配 否 工具按稳定排序自动分配，例如源 Actor 路径、组件名、material slot、材质路径组成排序键，避免每次构建 index 随机变化。
写入顶点数据 否 Geometry Merge / Proxy 生成阶段写入 TexCoord7.x，同时保留原始 UV0 用于正常贴图采样。
贴图 slice / UVRect 否 TextureBatch 阶段自动生成，并写入 MappingData 或 PropertyTextureArrayInfo。
材质参数行 否 MaterialPropertyBatch 阶段读取 MI 参数，每个 materialIndex 对应 _PropTexture 一行。
批处理范围控制 默认否，可选是 默认可按关卡、Folder、Data Layer 或选区自动收集；当美术需要排除动态物/特殊表现时，再用 UI、Folder、Tag 或 Group 覆盖。

5.3 _PropTexture 的意义

_PropTexture 是“材质参数数据库”。它把原本每个 MI 上不同的 scalar/vector 参数变成 GPU 可索引的纹理表。附件里的系统每个材质占一行，每列是一组属性，甚至用 half packing 把多个半精度参数塞进 float4。

原始材质实例数据批处理后数据UE 方案含义
Metallic、Cutoff、Smoothness、BumpScale 等 scalar _PropTexture 行/列 做 T_PropTexture_Cluster，按 material index 读取。
颜色、强度、UV ST 等 vector _PropTexture 或参数表 优先用 PropertyTexture；少量全局参数保留为材质参数。
Static Switch 不能直接放入属性表 必须拆 batch variant，或改造成运行时分支/参数。

5.4 Texture2DArray 的意义

Texture2DArray 解决的是“同一张 shader 里如何访问多个原始材质的贴图”。每个材质的 BaseColor、Normal、ORM 等贴图被打进数组 slice。若原始贴图尺寸不同，还需要 UVRect 或 atlas rect，告诉 shader 这个材质内容位于 slice 的哪个区域。

originalUV
  -> frac / tiling
  -> uvRect.xy + originalUV * uvRect.zw
  -> Texture2DArray.Sample(sliceIndex, mappedUV)

这也是为什么 2DArray 方案不是简单把贴图丢进去就结束：它需要 slice index、UVRect、贴图通道规范、尺寸/格式处理和材质参数表配套。

5.5 它为什么能 1 Draw Call

EnvLitBatch 能做到 1 Draw Call，必须同时满足以下条件：

条件原因UE 对应任务
Mesh 已经离线合并 不同物体仍独立提交，就不会变成一个 draw call。 ProxyMesh / Geometry Merge / HLOD。
同一个 shader 变体 不同 static switch / render state 会拆 draw call。 材质兼容审计、变体收敛。
每个顶点有 material index 合并后 shader 仍要知道每个像素该用哪个材质数据。 TexCoord7.x / vertex color / custom attribute。
贴图已打包 不能每个 MI 绑定独立贴图。 T2DA 或 VT Atlas。
参数已打包 不能每个 MI 保留独立 scalar/vector 参数。 PropertyTexture / 参数表。

5.6 与 VT/RVT 的本质区别

系统驱动量解决问题不能自动解决
EnvLitBatch / T2DA Batch 离线写入的 batchIndex/materialIndex 任意静态道具/建筑的材质属性与贴图索引。 贴图 streaming 和大地面多层缓存。
Streaming VT 虚拟纹理坐标和页表 大纹理按需加载、显存/streaming 管理。 材质参数表、mesh 合并、draw call 合并。
RVT 世界位置 / RVT 写入读取 地面、贴花、污渍、混合层缓存。 任意物体的材质 ID、PBR 参数索引、静态 mesh 合并。
关键判断：VT/RVT 可以作为 batch material 的“采样资源层”，但不能替代 EnvLitBatch 的“材质属性索引层”。如果用 VT Atlas 代替 Texture2DArray，也仍然需要 materialIndex、UVRect、PropertyTexture、合并 mesh。

5.7 项目实际采样方式边界

根据当前完整代码资料，项目里的建筑/道具 EnvLit 系和地形 VT 系是两套不同采样逻辑，不能混在一个概念里。

材质类型采样驱动量采样形式是否使用 VT 逻辑合批含义
普通道具/建筑 EnvLit 顶点 UV0 SAMPLE_TEXTURE2D(_BaseMap, sampler_BaseMap, uv) 否 每个材质球独立 Texture2D；不会因为 VT 自动合批。
合批道具/建筑 EnvLitBatch 顶点 UV0 + per-vertex batchIndex SAMPLE_TEXTURE2D_ARRAY(_BaseMap, ..., uv, sliceIndex) 否 贴图在 Texture2DArray，材质 ID 来自 batchIndex；这是建筑/道具合批核心。
地形/地面 VT Shader 世界坐标 / VT page lookup _VTLookupTex 查页表，再采样 _vt_diffuse_tex 等数组页。 是 服务连续大面积地面纹理、分页和缓存；不提供 EnvLitBatch 的材质属性表。 
// 地形/地面 VT 逻辑示意
float4 page = LOAD_TEXTURE2D(_VTLookupTex, loadpos) * 255;
float2 inPageOffset = frac(uv * exp2(_VTPageParam.z - page.b));
half4 albedo = SAMPLE_TEXTURE2D_ARRAY_LOD(_vt_diffuse_tex, ..., inPageOffset, texindex, 0);

// EnvLitBatch 逻辑示意
batchIndex = ReadVertexMaterialIndex();
prop = LOAD_TEXTURE2D(_PropTexture, int2(propertyColumn, batchIndex));
sliceIndex = ReadTextureSliceIndex(batchIndex);
color = SAMPLE_TEXTURE2D_ARRAY(_BaseMap, ..., uv0, sliceIndex);
项目规则更新：EnvLit / EnvLitBatch 当前都没有 _VTLookupTex、世界坐标分页、VT page table 逻辑。VT 目前应归入地形/地面系统；建筑/道具合批仍以 Texture2DArray + batchIndex + PropertyTexture + 合并 Mesh 为主线。

5.8 对 UE5.8 当前方案的落地翻译

我们的 UE 方案应拆成两个互补路径：

路径适合对象核心数据优先级
T2DA Batch 建筑墙体、环境模块、贴图规格较统一的静态 cluster。 TexCoord7.x materialIndex、Texture2DArray、PropertyTexture。 先做，用来复刻 EnvLitBatch 的确定性。
VT Atlas Batch 大量 unique 贴图、尺寸差异大、环境贴图希望统一 VT 管线的资产。 materialIndex、VT atlas rect、PropertyTexture。 第一阶段可作为主线；T2DA 作为 fallback，不要求美术手动制作 atlas。
RVT Ground 地面、贴花、污渍、墙根融合、俯视角大面积材质层。 RVT 写入材质、RVT 采样材质、地面质量分级。 与建筑 batch 并行，但逻辑独立。

5.9 新增实施要求

  1. M_Env_Building_Batch 必须明确 materialIndex 输入，建议先用 TexCoord7.x。
  2. MaterialBatchBuild 生成 ProxyMesh 时，要把每个源 material slot 对应的 materialIndex 写入合并后顶点。
  3. 生成 T_PropTexture 时，要固定列布局，例如 BaseMap_ST、Normal_ST、Metallic、Smoothness、Cutoff、BumpScale、材质灯预算等。
  4. 生成 T2DA 时，要记录每个材质每个通道的 slice index 和 UVRect。
  5. 材质审计要把 static switch 不一致、贴图通道缺失、尺寸/格式不兼容列为可解释拒绝原因。
  6. VT/RVT 只作为地形/地面或独立采样/缓存策略进入方案，不再写成“可以自动材质合批”。
  7. 如果未来研究建筑 VT Atlas Batch，必须另建一套 EnvLitBatch-like 索引层；不能直接复用地形 VT 的世界坐标页表逻辑。

6. 美术侧应该怎么参与

我不会要求美术手动理解 batchIndex、Texture2DArray slice、PropertyTexture 行列、UVRect 或 MappingData。理想状态下，美术实际操作是 0：正常摆场景，使用正确的 EnvLit / Env 系父材质即可。只有当工具需要知道“哪些物体应该进入同一个批、哪些必须排除”时，美术才通过 Folder、Data Layer、选区、Tag 或插件窗口表达范围意图。

美术边界：美术不手填 batchIndex，不手动指定 slice，不手改顶点 UV 承载材质 ID。所有这些都由 MaterialBatchBuild / EnvBatch 工具链在生成合并 Mesh 和批材质资源时自动完成。

6.2 推荐美术流程

2. 自动扫描工具收集使用 EnvLit 系材质的静态 Mesh。
3. 可选分组需要控制范围时，再用 Folder、Data Layer 或 Level Instance。

运行时不建议在玩家当前视野里硬切整个房间的 Source/Proxy。更合理的是在关卡加载、房间切换、流送边界或重新进入区域时生效。

档位运行时显示说明

Steam Deck 15W / Switch Candidate Source + Proxy 混合 近景保留 Source，中远景或重复区域用 Proxy。

9. 俯视角场景、ISM/HISM、LOD 与 RVT/VT 追加研究

本节结论：俯视角项目确实会弱化“远景 LOD”的收益，但不会取消 LOD 的价值。重复模型优先使用 ISM/HISM；非重复静态环境仍保留 Geometry Merge / Proxy / HLOD 类合批。地面与大面积表面建议以“离线烘焙 + 运行时流式加载”的 Baked SVT 为低功耗目标；UE 运行时 RVT 适合编辑期预览、高档动态效果或局部验证，但不应默认作为掌机/手机最终运行时方案。VT/SVT/RVT 都不等于 UE 自动合并 draw call。

9.1 重复模型：ISM/HISM 仍是第一优先

如果同一个 mesh 被大量重复使用，例如石块、栏杆、柱子、碎砖、重复装饰件，优先走 ISM 或 HISM。这比先做 Geometry Merge 更稳，因为它保留实例变换、编辑成本低，也更适合美术反复调整。

情况建议原因
同 StaticMesh、同材质、大量重复 ISM/HISM 减少提交成本，保留单个 mesh 的 LOD/剔除逻辑，编辑和回退方便。
同区域内不同 mesh，但都是静态背景 Geometry Merge / ProxyMesh ISM/HISM 无法跨不同 mesh 合批，需要代理网格或 HLOD。
近景重复物，轮廓很重要 ISM/HISM + LOD 避免 Proxy 破坏近景质量。
中低配、远离镜头中心、不可交互环境块 ProxyMesh / HLOD / Baked 替代 降低 draw call、section、材质复杂度。

9.2 俯视角是否还需要 LOD

需要，但 LOD 策略要从“远景透视”改成“屏幕占比、遮挡、平台档位”。俯视角可能没有传统远景地平线，但仍有：

  • 镜头边缘的小物体屏幕占比下降。
  • 楼层、墙体、地面装饰在低分辨率/掌机屏幕上细节不可见。
  • 大量重复小件即使不远，也会造成 vertex、材质和 draw call 压力。
  • 掌机 5W 档需要更激进地降低三角面和材质采样。

因此 LOD 不应取消，只是优先级变化：重复件保留 mesh LOD；房间/走廊 cluster 在 Medium/Low 用 Proxy/HLOD；PC Ultra 继续保留 Source。

9.3 俯视角地面：Baked SVT 是低功耗目标，RVT 是制作/动态辅助

你的判断基本正确：俯视角下地面占屏幕比例高，地面材质、贴花、污渍、混合层、角色/环境接触痕迹会很重要。但这里需要拆清楚两种路线：UE 默认 RVT 会在运行时把写入物体渲染到虚拟纹理页面；你贴出来的项目 VT 更接近 Baked SVT，页面内容在离线 CI 或构建阶段烘好，运行时只做反馈、异步加载、纹理上传和页表更新。

地面效果Baked SVT 适配度说明
地表多层混合 高 可将复杂层混合结果缓存，减少主地面材质实时分支。
贴花、污渍、破损、边缘融合 高 静态 decal-like 效果应在离线阶段烘入 SVT tile，运行时避免大量透明/贴花叠加。
地面与墙根/道具融合 中高 适合做高度/颜色/法线类混合，但要规范写入通道。
高频动态痕迹 中 能做，但不适合 Baked SVT 主路径；动态内容应走局部 decal、局部 RVT 或单独效果系统。
角色阴影/实时交互 低到中 Baked SVT/RVT 更适合缓存材质数据，不应替代实时阴影和关键交互表现。

9.3.1 RVT 采样后还是 RVT，还是变成 SVT

结论：UE 的 RVT 默认仍是运行时虚拟纹理：场景物体写入 RVT，地面/墙面材质再采样 RVT。它不是自动变成普通 SVT。你贴出来的项目 VT 不是这种默认 RVT，而是“Baked SVT”：离线把 diffuse、AO、shadow、local shadow 等层烘成 tile，运行时通过 feedback / LOD 选择页面，从包体异步加载并上传到物理缓存，再更新 lookup table。
模式运行时行为成本建议用途
纯 RVT 运行时按页渲染写入 RVT，材质采样 RVT。 有页面渲染、更新、反馈、物体写入成本；动态变化越多越贵。 PC、编辑期预览、需要动态地表变化的区域。
RVT + Streaming Low Mips 低 mip 预构建为 streaming virtual texture，高 mip/近处仍可由 RVT 动态生成。 降低远处/低精度页面运行时生成压力；仍保留近处动态质量。 Steam Deck 15W、Switch Candidate、一般掌机档。
Streaming Mips Only 使用已构建的 streaming texture，不再依赖运行时页生成。 运行时最稳，但动态写入/实时变化能力下降。 Steam Deck 5W、手机、低功耗/低画质档，适合静态地面与墙根效果。
Baked SVT 离线烘焙页面；运行时按相机 feedback/LOD 异步加载 tile，上传物理缓存并更新页表。 无运行时 RVT 渲染 pass；主要成本是 IO、解压/上传、页表更新和物理缓存容量。 项目地形/地面/大面积静态表面的低功耗目标方案。

UE 侧可参考的关键设置是 URuntimeVirtualTextureComponent::StreamingTexture、StreamLowMips、bUseStreamingMipsOnly，以及控制台变量 r.VT.RVT.StreamingMips 和 r.VT.RVT.StreamingMips.UseAlways。但项目目标不应只停在“运行时 RVT”，而应评估能否建立类似 Baked SVT 的离线 tile 生产与运行时页面调度。

因此地面/墙面建筑效果推荐拆成两层：美术制作和高档动态验证阶段可以用 RVT 或 decal 表达复杂混合；发布低功耗档时尽量把静态结果烘成 Baked SVT / Streaming VT 页面。这样既保留制作灵活度，又避免掌机/手机运行时频繁渲染 RVT 页面。

9.3.2 地形、墙根、建筑表面推荐策略

  1. 主地面：低功耗目标是 Baked SVT，离线烘焙 BaseColor / Normal / Roughness / AO / shadow 等项目需要的最小通道集，不要一次缓存过多属性。
  2. 墙根、地面污渍、贴花、融合层：静态内容优先离线烘入 Baked SVT；需要动态更新的内容才进入局部 RVT 或 decal。
  3. 静态建筑墙面：不要为了墙面统一全部走 RVT；如果是合批建筑，仍优先走 Selective VT Atlas Batch。只有墙根融合、污渍投射、地表交界这类效果才读 RVT。
  4. 掌机/手机档：可以开启 VT 支持，但应减少运行时 RVT 动态写入；优先使用 Baked SVT 或 UE 的 Streaming Texture / bUseStreamingMipsOnly 类降级。
  5. 动态效果：脚印、破坏、实时液体等高频更新不要默认写进全局大 RVT，应该单独评估更新频率和范围。

9.3.3 墙面材质混合、贴花与 XYZ RVT

结论：墙面可以做 RVT 投射，但不建议默认做“XYZ 三方向全局 RVT”。UE 的 RVT 本质仍是二维虚拟纹理页面加体积投射，不是低成本的三维体纹理。XYZ 方案更像三套方向不同的 2D RVT：XY 管地面，XZ/YZ 管墙面。它可以解决局部墙面投射问题，但会带来更多 RVT volume、更多 page table / tile、更多材质采样和墙角接缝处理。
墙面需求推荐方案原因
静态墙面花纹、污渍、颜色混合 优先在批处理构建时 bake 到 Selective VT Atlas / 批材质贴图 这是最适合合批的路径：运行时不增加 RVT 写入，不增加 decal pass，也能和 ProxyMesh / BatchMaterial 一起提交。
墙根、地面到墙面的脏污过渡 读取地面 RVT 或局部墙根 RVT 这类效果通常是低频大面积融合，适合通过一层缓存减少多层贴花和复杂混合。
墙面海报、涂鸦、裂痕、局部破损 静态内容用 decal card / 小 mesh / 小图集，最终可 bake 进批贴图；动态内容保留局部 decal 不要为了少量局部图案让整面墙长期采样三方向 RVT。静态内容进离线合批，动态内容按范围控制。
房间级墙面投射、程序化大面积墙面混合 可做局部 Wall RVT 实验，例如按房间或墙面 cluster 建 XZ/YZ RVT 只在明确需要世界空间投射、且普通贴花/批贴图难以维护时使用。低功耗档应构建 Streaming Texture 或降级为 baked atlas。
全场景 XYZ 三方向 RVT 不作为默认方案 每个材质默认采样三套 RVT 会放大 GPU 成本和内存压力；墙角、法线方向、投射重叠、Streaming mips 都需要额外规则。

推荐把墙面分成“静态可烘”和“运行时需要投射”两类：静态可烘的墙面混合和贴花进入 EnvBatch 构建流程，由工具在生成 VT Atlas / fallback Texture2DArray 时自动合成；运行时需要投射的效果才使用局部 RVT 或 decal。这样美术仍然可以在场景里摆效果，但最终构建时由工具决定是 bake 到批材质，还是保留为运行时投射。

  1. 默认墙面：使用正常 EnvLit / EnvLitBatch 材质，静态效果进入批处理贴图缓存。
  2. 墙根融合：静态结果优先烘进 Baked SVT；需要运行时变化时才读取地面 RVT，一般控制为一次 RVT 采样。
  3. 局部墙面投射：建立 WallRVT 或 RoomRVT 分组，只覆盖指定房间或墙面 cluster。
  4. XYZ RVT：只作为专项测试档，材质根据法线主轴选择一个方向采样，避免每个像素同时采三轴。
  5. 掌机/低功耗档：优先把静态墙面效果 bake 到 VT Atlas 或 Baked SVT；局部 RVT 构建 Streaming Low Mips，必要时关闭动态写入。

9.3.4 Baked SVT 与建筑批材质是否冲突

结论：不冲突，也不是重复加载。Baked SVT 主要服务地面/地形/大面积世界空间表面；EnvLitBatch / Texture2DArray / VT Atlas 主要服务建筑、道具、环境物件。两者使用不同几何体、不同 UV 来源、不同 shader 和不同 GPU 资源。
系统对象采样依据运行时资源主要风险
Baked SVT 地面、地形、大面积静态表面 世界坐标、页表、LOD feedback 物理缓存、lookup table、离线 tile 包 物理缓存太小会换页卡顿，太大会浪费显存；IO/上传峰值需要限流。
EnvLitBatch 建筑、道具、静态环境物件 顶点 UV + per-vertex materialIndex Texture2DArray 或 VT Atlas、PropertyTexture、ProxyMesh Texture2DArray/Atlas 是批次全量驻留；batch group 太大会显存膨胀。
局部 RVT 动态地表、墙根、房间级特殊投射 世界坐标或局部投射空间 RVT volume、运行时页面、写入 pass 运行时渲染 pass 和动态更新成本；低功耗档必须严格限制范围。

因此真正要监控的是总显存预算，而不是“是否重复”：总显存约等于 Baked SVT 物理缓存 + EnvLitBatch 的 Texture2DArray/VT Atlas + 普通 unique 贴图 + 阴影/后处理等其他渲染资源。低功耗档应同时限制 SVT physical cache capacity、批次贴图尺寸、单批材质数量和动态 RVT 数量。

9.4 VT/RVT 是否能“自动材质合批”

重要边界：不能把 VT/RVT 理解成 UE 自动 draw call 合批。Streaming Virtual Texturing 解决的是纹理按需加载和显存/streaming；Runtime Virtual Texture 解决的是把场景材质结果写入/读取虚拟纹理。它们不会自动把不同 StaticMeshActor、不同材质 slot、不同 mesh section 合成一个 draw call。

VT/RVT 能帮到材质合批的地方是“统一采样来源”和“减少材质复杂度”，例如地面所有局部混合都变成采样一张 RVT，而不是每个 decal/材质层单独渲染。但 draw call 是否减少，还取决于：

  • 是否减少了 decal/mesh pass。
  • 是否减少了材质 slot 和 mesh section。
  • 是否使用了 ISM/HISM、Geometry Merge、HLOD 或 ProxyMesh。
  • 是否把多个材质实例收敛到同一批处理材质变体。

9.5 推荐的新分工

对象类型优先方案是否需要 LOD是否用 RVT/VT
重复小模型 ISM/HISM 需要，尤其掌机档。 贴图可用 VT，但不是核心合批手段。
非重复静态建筑块 Geometry Merge / ProxyMesh / HLOD Source 有 LOD，Proxy 按档位简化。 可结合 Texture2DArray/VT 管理 unique 贴图。

地面贴花/污渍/混合效果 静态内容烘入 Baked SVT；动态内容写入局部 RVT 或 decal 按贴花载体决定。 强烈建议缓存静态结果，能降低叠加材质成本。
动态交互/可破坏物 独立 Actor，不进静态合批 按资产距离和屏幕占比。 可读取 RVT 做融合，但不应被 RVT 合并逻辑吞掉。

9.6 对当前方案的调整

  1. 模型侧：重复件优先 ISM/HISM，不急着 Geometry Merge；非重复静态环境才进入 EnvBatch/ProxyMesh。
  2. LOD 侧：保留 LOD，但按俯视角屏幕占比和平台档位重新设阈值，不按传统远景逻辑硬套。
  3. 地面侧：单独建立 Baked SVT 地面管线，允许美术做多层混合、贴花、污渍、边缘融合，然后在构建阶段烘成静态页面；RVT 只作为制作预览、PC 高档或局部动态辅助。
  4. 材质合批侧：RVT/VT 作为降低采样和 streaming 压力的手段，不替代 Geometry Merge、ISM/HISM、HLOD。
  5. 验证侧：需要单独增加 Ground_BakedSVT_Only、Ground_RVTPreview_Only 和 Ground_BakedSVT + Batch 测试。

10. UE5.8 Material Cache 与 MaterialCacheVirtualTextureTag 研究

本节结论：MaterialCacheVirtualTextureTag 是 UE5.8 引擎内部/实验性的 Material Cache 虚拟纹理标签资产，用来描述“缓存哪些材质属性、如何压缩打包、tile 数量倍率”等信息。它不是普通 Streaming VT，也不是 EnvLitBatch 式的自动 draw call 合批开关。

10.1 源码位置与项目现状

当前项目源码中没有直接使用 MaterialCacheVirtualTextureTag。它来自本机 UE5.8 引擎源码：

D:\UE\UE_5.8\Engine\Source\Runtime\Engine\Internal\MaterialCache\MaterialCacheVirtualTextureTag.h
D:\UE\UE_5.8\Engine\Source\Runtime\Engine\Internal\MaterialCache\MaterialCacheVirtualTexture.h
D:\UE\UE_5.8\Engine\Source\Runtime\Engine\Internal\MaterialCache\MaterialCacheAttribute.h
D:\UE\UE_5.8\Engine\Source\Runtime\Engine\Public\Materials\MaterialExpressionMaterialCache.h

项目当前 Config/DefaultEngine.ini 里 r.VirtualTextures=False。因此 Material Cache、RVT、Streaming VT 这类路线都不能视为已经启用，必须先进入平台分级和功能验证。

10.2 MaterialCacheVirtualTextureTag 描述什么

字段 / 概念含义对我们的意义
Attributes 决定缓存哪些材质属性，例如 BaseColor、Normal、Roughness、Specular、Metallic、Opacity、WorldPosition、Displacement、Mask、Float。 类似“缓存通道清单”。地面/大面积静态材质可以只缓存必要属性，掌机档减少无用层。
TileCountMultiplier 对使用该 Tag 的虚拟纹理 tile 数做倍率调整，运行时实际 tile 数为 TileCount * TileCountMultiplier。 可以按材质重要性调整缓存分辨率。地面主视觉可高一些，远处/低档可低一些。
bEnableCompression 是否对缓存层启用压缩。 影响显存和质量。掌机优先压缩，PC 可按视觉需求提高质量。
bEnableBaseColorHQ BaseColor 使用更高质量的 YCoCg 存储，但显存压力更高。 地面或近景建筑可测试；低功耗档不宜默认打开。
bEnableNormalHQ Normal 使用更高质量的独立 8:8 octahedron 存储。 法线细节强的地面可测试；大量普通墙面不应盲目高质量。
RuntimeLayers 根据 Attributes 和压缩选项生成的运行时层布局。层数不能超过 VT 最大层数限制。 不是属性越多越好。属性越多，层越多，内存和采样压力越高。

10.3 它和 EnvLitBatch / VT / RVT 的关系

方案核心索引解决什么是否自动降低 draw call
EnvLitBatch per-vertex batchIndex + Texture2DArray slice + PropertyTexture row 把多个静态物件/材质离线合成同一批材质和合并 mesh。 是，但前提是 Geometry Merge、材质索引表和 batch shader 都完成。
Streaming VT 虚拟纹理页表 / tile streaming 贴图按需加载、降低显存峰值和 streaming 压力。 否。
RVT 世界位置 / RVT 写入读取 把地面混合、贴花、局部效果缓存为可采样材质结果。 不直接降低 mesh draw call，但可减少贴花/复杂材质叠加成本。
UE Material Cache MaterialCache 表达式 + MaterialCacheVirtualTextureTag + primitive/component 绑定 把指定材质属性缓存到虚拟纹理层，之后由材质采样缓存结果。 否。它偏向材质结果缓存，不替代 Geometry Merge / ISM / HISM。

10.4 启用条件和风险

  • 源码里 r.MaterialCache.Support 默认是 false，且是 ReadOnly；r.MaterialCache.Enabled 只有在 Support 与平台 VT 能力都满足时才有效。
  • UMaterialExpressionMaterialCache 需要材质图显式参与，材质缓存数据会记录到 MaterialCacheTags。
  • UStaticMeshComponent 有实验性字段：bWritesToMaterialCache、MaterialCacheTileCount、MaterialCacheUVRegion、MaterialCacheUVCoordinateIndex。
  • MeshPartition 插件也在使用 Material Cache，但路径位于 Engine\Plugins\Experimental\MeshPartition，不能直接按正式生产功能依赖。
  • Material Cache 需要 VT 支持，因此 PC、Steam Deck、Switch Candidate 都要单独验证。掌机平台要重点看 tile 内存、页表、cache miss、shader 编译和运行时抖动。

10.5 对当前项目的建议

  1. 短期不要用 Material Cache 替代 EnvLitBatch。建筑/道具合批仍以 Texture2DArray + PropertyTexture + batchIndex + Geometry Merge 为主线。
  2. Material Cache 可以作为 Phase 2.5 的研究项，用一个小型地面或静态墙面材质验证“缓存 BaseColor/Roughness/Normal 是否能降低复杂材质成本”。
  3. 如果开启 VT/RVT，先拆清楚三类资产：地面 RVT、建筑道具 EnvLitBatch、普通 unique 贴图。不要为了统一 VT 而把所有材质强行改成 Material Cache。
  4. 低功耗档的 Tag 应尽量少属性、低 tile 倍率、开启压缩；PC 档可以测试高质量 BaseColor/Normal。
  5. 只有当实测证明材质复杂度是瓶颈，且 Material Cache 在目标平台稳定，才把它纳入美术正式管线。
当前判断：Material Cache 更适合归入“材质结果缓存 / 地面或大面积静态表面优化”研究，不应归入“自动材质 draw call 合批”。我们的合批核心仍是减少 mesh section、减少材质 slot、ISM/HISM、Geometry Merge、HLOD/Proxy，以及 EnvLitBatch-like 的材质索引层。

10.6 美术是否可以不手动制作 Array

可以，而且应该这样做：美术不应该手动制作 Texture2DArray，也不应该手动维护 VT Atlas。美术侧继续在普通 EnvLit / Env 系材质实例里给 BaseColor、Normal、ORM、Mask 等正常贴图；批处理工具在 dry-run / apply 阶段自动读取这些贴图和参数，再生成批处理运行时需要的资源。
对象美术怎么做工具怎么做是否需要美术手动维护
原始材质实例 正常指定贴图和参数。 读取 Texture 参数、Scalar/Vector 参数、Static Switch、渲染状态。 否。
Texture2DArray 不制作。 按 batch group 自动生成 T2DA_BaseColor、T2DA_Normal、T2DA_ORM 等。 否。
共享 VT Atlas / Material Cache 不制作。 第一阶段即可作为主线：工具只对合批候选生成共享 VT/cache 和每个 materialIndex 的 UVRect / page 信息，不强制所有项目贴图转 VT。 否。
PropertyTexture 不制作。 自动把每个材质实例的参数写成按行索引的属性表。 否。
batchIndex 不填写。 模型 merge 时自动写入 TexCoord7.x 或等价顶点属性。 否。

10.6.1 开启 VT 是否是自动合批保证

不是。开启 VT 只能让项目具备虚拟纹理能力，并让特定贴图或生成资产进入虚拟纹理 streaming / page 管理体系，不能保证模型 merge 后自动合批。合批是否成立取决于工具是否生成了同一套批材质资源，并且所有候选物体满足同一 shader variant、同一 render state、可合并 mesh、可写入 materialIndex、可生成 PropertyTexture 和贴图映射。

因此更准确的表述是：

美术正常给贴图
  -> 工具自动收集材质
  -> 工具自动生成共享 VT Atlas 或 T2DA fallback
  -> 工具自动生成 PropertyTexture
  -> 工具自动写 batchIndex
  -> 工具自动生成 ProxyMesh / BatchMaterial
  -> 合格对象可合批；不合格对象进入拒绝报告或拆 variant

10.6.2 当前推荐生产策略

  1. 第一阶段：允许项目开启 VT/RVT 支持，但采用选择性策略。原始贴图不全量转 VT；只对进入合批的候选自动生成共享 VT Atlas。
  2. Texture2DArray 改为 fallback：如果某个平台、某类材质或某个批次不适合 VT Atlas，再回退到 T2DA 生成路径。
  3. 无论 T2DA 还是 VT Atlas，美术都不手动维护合批资源；合批资源全部生成在 /Game/Generated/MaterialBatch 或指定生成目录。
  4. 工具只能“保证合格候选会按规则生成批处理资产”，不能保证所有场景物件都进入同一个 batch。动态物、特殊材质、不同渲染状态、特殊 WPO/Masked/TwoSided、材质灯特殊路径都可能被拆分或拒绝。

10.6.3 合批 VT Atlas 的自动化边界

问题处理方式
是否需要所有原始贴图都开启 VT 不需要。项目可以开启 VT 支持，但工具只把生成的批资源设为 VT；原始美术贴图按资产规范保留。
是否需要美术把贴图放进 atlas 不需要。工具读取材质实例引用的普通贴图，自动生成合批专用 VT Atlas。
VT Atlas 能否直接替代 batchIndex 不能。VT Atlas 只替代 Texture2DArray 的采样资源层；batchIndex、UVRect、PropertyTexture、ProxyMesh 仍然必须生成。
是否能保证 merge 后自动合批 能保证“通过审计的候选”生成统一批材质和合并 mesh；不能保证所有物件都合批，拒绝原因必须写入报告。

11.3 决策标准

  • 如果 Batch Only 明显降低 draw call 但 GPU ms 不降，说明瓶颈可能不是 draw call，要转向材质/光照/像素成本。
  • 如果 Lumen Lite 成本高于合批收益，掌机默认不启用 Lumen Lite。
  • 如果 Proxy 视觉差异太大，只能用于 Low 或远景，不能进入 Medium。
  • 如果 Texture2DArray 导致贴图质量或 streaming 问题，需要调整尺寸规范或改用 VT/SVT 辅助；如果 Baked SVT 换页卡顿，需要调小 tile 上传峰值或增大物理缓存。
