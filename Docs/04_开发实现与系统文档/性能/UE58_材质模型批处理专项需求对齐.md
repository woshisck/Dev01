# UE5.8 材质与模型批处理专项需求对齐

更新时间：2026-06-29

本文用于继续推进“材质、模型、Geometry Merge、Texture2DArray、PropertyTexture、EnvBatch Tag、性能分级”的专项工作。它不是最终实现说明，而是下一轮实现前的需求边界和任务拆分。

## 1. 当前结论

- 性能分级、玩家画质设置、Lumen Lite 开关、DeviceProfile、profiling commandlet 已经有第一版。
- 材质和模型批处理仍处于专项前置阶段：已有审计、dry-run、partial apply 样板，但还没有把真实关卡批量替换成最终代理网格。
- VT/SVT 主要解决 unique 贴图 streaming 和显存压力，不等价于 draw call 合批。
- 真正节省 draw call 需要 Geometry Merge / HLOD / ISM-HISM / 批处理材质共同工作。
- 当前项目的批处理方向应优先做“静态环境 cluster 离线合并 + 每顶点/UV 携带 material index + Texture2DArray + PropertyTexture”。

## 2. 目标场景

第一轮代表关卡使用：

```text
/Game/Art/Map/Map_Data/L1_CommonLevel_corridor_S_Dungeon/L1_CommonLevel_corridor_01b.L1_CommonLevel_corridor_01b
```

第一轮目标不做全关卡替换，只做一个可审计、可回退、可对比的 corridor cluster。

## 3. M_Env_Building 需要专项处理的问题

目标材质：

```text
/Game/Art/Material/EnvMaterial/Main/M_Env_Building.M_Env_Building
```

已知审计结论：

- 材质是 `Masked / DefaultLit / No WPO / One-sided`，适合进入静态环境候选。
- 已经有 `T_Array_A`、`T_Array_M`、`T_Array_N` 三个 Texture2DArray 参数。
- 仍有 unique 贴图参数：`T Unique Color Map`、`T Unique MRA Map`、`T Unique Normal Map`、`T Unique Mask Map`。
- `Tex_LightInfo` 属于赛璐璐材质灯光/CanvasRenderTarget2D 类特殊输入，不应直接并入 BaseColor/Normal/ORM 的 Texture2DArray。
- Static Switch 会拆批，尤其是 `Use StainMat`、`Use Global Mask`、`Use UniqueMask Map`、`UseFakeLightInfo`、`Use CheapTri-Planar`。

专项目标：

1. 做一个批处理母材质或材质函数路径，例如 `M_Env_Building_Batch`。
2. 从 `TexCoord7.x` 或约定 UV 通道读取 material/slice index。
3. 用 index 读取 `_PropTexture` 对应行的 scalar/vector 参数。
4. 用 index 采样 `T_Array_A/N/M` 对应 slice。
5. `Tex_LightInfo` 先保留独立质量分级路径，由 `r.Yog.MaterialLightQuality` 和 `r.Yog.MaterialLight.MaxLightInfoCount` 控制采样预算。
6. 将 Static Switch 组合收敛为少量可批处理变体，不能把所有 MI 不加审计地混到同一个 batch。

## 4. 模型与贴图处理规则

静态环境模型要进入批处理，建议满足：

- Actor/Component 确认为静态，不参与交互、破坏、拾取、剧情脚本移动。
- StaticMeshComponent mobility 应为 Static 或经过关卡侧确认可转静态。
- 材质父级一致或可映射到同一个 batch shader。
- Static Switch/keyword 组合一致。
- 贴图通道集合一致，例如 BaseColor、Normal、ORM、Mask。
- Texture2DArray slice 需要统一尺寸、格式、mip 策略。
- Unique 贴图可以被打包成 Texture2DArray slice，但必须记录 slice index 和 UV/ST 参数。
- 不适合合并的物体应打 `EnvBatch.Exclude`。

不建议进入第一轮批处理：

- 门、机关、宝箱、可破坏物、拾取物、敌人、武器、VFX、需要独立碰撞/脚本事件的物件。
- 近景英雄资产或需要独立材质变化的重点资产。
- 高频动态光照交互对象。

## 5. 自动化流程目标

长期目标是“打 tag 后自动 dry-run，人工审查后 partial apply”，不是直接全量替换。

推荐流程：

1. 美术或关卡设计在编辑器里用 EnvBatch Tagger 给候选 Actor 打 tag。
2. 跑 `MaterialBatchBuild` dry-run，生成候选、拒绝原因、贴图计划、合并计划。
3. 人工审查拒绝原因，尤其是 `DynamicMobility`、材质不兼容、贴图尺寸不一致。
4. 只对通过审查的 cluster 跑 partial apply：
   - `-ApplyMappingOnly`
   - `-ApplyTextureArraysOnly`
   - `-ApplyPropertyTextureOnly`
   - `-ApplyProxyMeshOnly`
   - `-ApplyBatchMaterialOnly`
5. 在测试关卡中手动或脚本切换 Source / Proxy。
6. 做 Baseline、Batch Only、Lumen Lite Only、Batch + Lumen Lite、Low Baked 的 GPU profile。
7. 数据通过后，再考虑更大范围推广。

## 6. 可交给后续 AI 继续处理的任务

优先级从高到低：

1. 读取 `M_Env_Building` 材质图，整理所有 texture/scalar/vector/static switch 参数表。
2. 设计 `M_Env_Building_Batch` 的输入约定：UV 通道、slice index、PropertyTexture 布局、LightInfo 降级路径。
3. 为 `MaterialBatchBuild` 增加材质兼容性报告：父材质、static switch、texture set、尺寸格式、拒绝原因。
4. 在 corridor 关卡中挑一个小 cluster，先人工确认 20-50 个 Actor。
5. 生成 Texture2DArray 和 PropertyTexture 后，在材质预览或测试 map 中做视觉 parity。
6. 再做 ProxyMesh 切换和 draw call / GPU ms 对比。

## 7. 第一轮验收标准

第一轮不要求全项目自动合批，要求：

- 一个 corridor cluster 有明确 Source tag。
- dry-run 报告可解释所有候选和拒绝项。
- 至少一个 batch 材质能正确通过 index 读取 Texture2DArray slice。
- ProxyMesh 与 Source 视觉差异可接受。
- `stat rhi` 或 RenderDoc/Unreal Insights 能看到 draw call 减少。
- 没有破坏主菜单、玩家画质设置、正常 Play 启动路径。
