# 卡牌设计总纲

## 设计背景

核心方案的卡牌系统用于替代旧 2D 背包在战斗中的前台表达。

玩家不再需要频繁停下来整理复杂背包，而是在战斗中通过一条 1D 卡牌序列获得构筑反馈：

```text
攻击 -> 消耗当前卡 -> 匹配动作 -> 激活特征 -> 连携转译 -> 终结爆发 -> 洗牌回填
```

本系统参考 Noita 的“可预测因果链”，但不追求无限法术拼装。目标是让玩家感到组合自由，同时让制作和代码保持可控。

## 核心设计思想

卡牌组合不靠隐藏配方，而靠有限字段和有限特征。

```text
卡牌 = 攻击要求 + 效果形态 + 默认效果 + 标志性特征 + 连携词条 + 终结技 + 护甲规则 + 表现反馈
```

```text
状态 = 默认效果 + 标志性特征
```

默认效果是状态本身的基础表现，例如燃烧每秒掉血、中毒按最大生命值掉血。

标志性特征是状态可以被连携读取和转译的独特能力，例如中毒的传染、燃烧的周围灼烧、撕裂的位移惩罚。

## 设计目标

1. 降低玩家理解成本
   玩家只需要理解当前卡、上一张特征和自己的攻击动作。

2. 提升战斗爽感
   卡牌服务命中、击退、状态、爆炸、终结技等即时反馈。

3. 控制制作复杂度
   所有卡牌都从有限字段中组合，不为每两张卡单独写配方。

4. 保留构筑惊喜
   特殊化学反应可以存在，但只作为稀有卡、特殊卡或特殊武器投放。

## 关键规则

- 动作不匹配时，只触发默认效果，不激活特征。
- 连携默认只读取上一张输出的 `TraitContext`。
- 当前效果形态必须能解释上一张特征，否则降级为数值增强或默认效果。
- 强控制、诅咒、强爆炸不默认传递。
- 护甲交互读取全局表，避免每张卡重复设计。
- 暂不考虑热度，后续只作为数值、范围和表现强化。
- 终结技通常单独设计，可以读取当前特征，但不强制走通用转译。

## 文档阅读顺序

1. `卡牌设计准绳.md`
   先确认制作红线和整体方向。

2. `卡牌字段规范.md`
   再确认单张卡允许填写哪些字段。

3. `卡牌特征库.md`
   查看状态、特征和护甲全局表。

4. `卡牌连携规则.md`
   查看特征如何读取、转译和降级。

5. `卡牌设计填写表.md`
   最后用模板填写单张卡或多卡组合。

## 一句话标准

每个设计最终都应该能说成：

```text
因为上一张卡提供了【特征】，所以当前卡的【效果形态】发生了【可见变化】。
```

如果这句话说不清楚，这张卡或这组连携就需要收窄。
