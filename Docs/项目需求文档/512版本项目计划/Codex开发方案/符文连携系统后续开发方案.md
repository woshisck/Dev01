# 符文连携系统后续开发方案

## 方案结论

本阶段暂停“所有卡牌都拥有无限连携”的设计，改为“少数连携卡提供明确机制变化”。普通卡牌保持稳定、直接、可预期；连携卡牌承担构筑亮点，通过读取前后卡牌信息产生不同效果。

本阶段不处理三连携和终结技，不要求战斗系统立刻改连招逻辑。符文系统先完成数据结构、解析规则、事件结果和配置文档；连招是否继续、是否退出连招状态由战斗系统判断并通知，背包/卡组/符文系统只消费这个结果。

## 旧方案问题

| 问题 | 影响 |
| --- | --- |
| 所有卡牌都可连携 | 设计量不可控，每张卡都需要前后关系、失败关系和 UI 解释，资产维护压力过大 |
| 普通卡依赖 Light / Heavy 匹配 | 玩家需要同时关注动作匹配和卡组顺序，注意力被拆散，不利于快节奏动作战斗 |
| `BaseFlow` / `MatchedFlow` 语义混用 | “普通触发”“动作匹配奖励”“连携成功效果”“断链效果”都挤在两个 Flow 字段里，后续难以配置 |
| `LinkMode = ReadPrevious / PassToNext` 过于粗糙 | 无法表达一张连携卡正向、反向、断链时分别触发不同效果 |
| `PendingLinkContext` 只存一张卡 | 无法稳定表达“当前处于连携阶段”“下一张是否接受连携”“断链原因是什么” |
| UI 事件只知道是否触发 Link | 无法向 HUD 解释是正向连携、反向赋能、还是断链普通释放 |

## 新核心规则

### 普通卡牌

- 只要攻击动作命中并触发出牌，普通卡就会被消耗并激活。
- 普通卡不再用 Light / Heavy 判断是否激活。
- 连击数或连招阶段只影响普通卡的倍率数值，不影响是否触发；倍率使用全局配置，不做每张普通卡单独倍率曲线。
- 普通卡可以接收来自前一张连携卡的反向赋能。

### 连携卡牌

连携卡是少数特殊卡牌，拥有可配置的正向连携、反向连携和断链释放。

| 类型 | 触发理解 | 示例 |
| --- | --- | --- |
| 正向连携 | 连携卡读取前一张已打出的卡，根据前一张卡的分类或标签触发自己的特殊效果 | 前一张是攻击牌时，月光触发效果 A |
| 反向连携 | 连携卡先正常释放，并把自身特性挂给后一张卡；后一张卡满足条件时触发单独配置的新效果 | 月光后一张是攻击牌时，后一张额外触发效果 B |
| 断链释放 | 战斗系统判定玩家退出连招状态，或下一张不满足条件时，连携卡不再等待，按普通效果打出 | 月光没有完成连携时，只触发月光的基础效果 |

### 动作与连携关系

- 动作类型不再决定普通卡是否激活。
- 动作类型只作为连携卡释放条件的一部分，用于判断某个连携是否成立。
- 玩家主要考虑“是否继续连招”和“卡牌前后顺序”，而不是每张普通卡是否匹配 Light / Heavy。
- 当战斗系统判定玩家退出连招状态，并且当前处于连携阶段时，才出现断链情况。

### 不在本阶段处理

- 三连携。
- 终结技。
- 所有卡牌泛化连携。
- 复杂多段缓存、跨洗牌保留连携、跨房间保留连携。

## 推荐数据结构调整

### 保留字段

| 字段 | 处理 |
| --- | --- |
| `bIsCombatCard` | 保留 |
| `CardType` | 保留，但建议 `Link` 只用于少数机制卡 |
| `BaseFlow` | 保留，语义调整为“卡牌正常释放效果” |
| `DisplayName` / `HUDReasonText` | 保留 |

### 弱化或废弃字段

| 字段 | 新语义 |
| --- | --- |
| `RequiredAction` | 不再用于普通卡激活；迁移为连携条件字段，或后续废弃 |
| `MatchedFlow` | 不再表示动作匹配奖励；建议改名或拆分 |
| `LinkMode` | 不足以表达正向/反向/断链，建议替换为更明确的连携配置 |
| `bRequiresComboFinisher` | 本阶段暂不扩展，终结技阶段再处理 |

## 运行时解析规则

### 普通卡

1. 攻击命中，卡组消耗当前卡。
2. 执行 `BaseFlow`。
3. 根据当前连击数或连招阶段读取全局倍率配置。
4. 如果存在来自上一张连携卡的反向赋能，并且条件满足，则额外执行 `BackwardEffect.Flow`，这是单独配置的新效果，不复用后一张卡原本的 `BaseFlow`。
5. 记录为 `LastResolvedCard`。

### 正向连携卡

1. 攻击命中，卡组消耗当前连携卡。
2. 读取 `LastResolvedCard`。
3. 如果前一张卡满足 `ForwardEffect.Condition`，执行 `ForwardEffect.Flow`。
4. 如果不满足，按 `BreakPolicy` 执行断链释放。
5. 正向连携结算后不继续无限传递。

### 反向连携卡

1. 攻击命中，卡组消耗当前连携卡。
2. 连携卡先执行自己的 `BaseFlow`。
3. 如果反向连携配置有效，生成一个 `PendingLinkContext`，等待下一张卡。
4. 下一张卡打出时，如果满足 `BackwardEffect.Condition`，额外执行 `BackwardEffect.Flow`。
5. 如果战斗系统通知退出连招状态、洗牌、换武器、或下一张不满足条件，则按 `BreakPolicy` 清理这次等待。

## 需要新增的运行时上下文

当前接口只有 `ActionType / bIsComboFinisher / bFromDashSave`，不足以表达新规则。符文系统可以先定义上下文，战斗侧后续逐步接入。连招状态的权威来源是战斗系统，卡组侧不做超时猜测。

```cpp
USTRUCT(BlueprintType)
struct FCombatDeckActionContext
{
    GENERATED_BODY()

    UPROPERTY(BlueprintReadWrite)

    UPROPERTY(BlueprintReadWrite)
    int32 ComboIndex = 0;

    UPROPERTY(BlueprintReadWrite)
    bool bComboContinued = true;

    UPROPERTY(BlueprintReadWrite)
    bool bExitedComboState = false;

    UPROPERTY(BlueprintReadWrite)
    bool bFromDashSave = false;

    UPROPERTY(BlueprintReadWrite)
    FGameplayTagContainer ComboTags;
};
```

## 结果事件建议

`FCombatCardResolveResult` 建议增加：

| 字段 | 用途 |
| --- | --- |
| `bTriggeredBaseFlow` | 普通释放 |
| `bTriggeredForwardLink` | 正向连携成功 |
| `bTriggeredBackwardLink` | 后向赋能成功 |
| `bPendingBackwardLink` | 当前卡创建了等待后一张的赋能 |
| `bLinkBroken` | 连携断链 |
| `LinkBreakReason` | 退出连招状态、下一张不满足、洗牌、换武器等 |
| `LinkedSourceCard` | 提供连携信息的卡 |
| `LinkedTargetCard` | 被连携影响的卡 |
| `AppliedMultiplier` | 本次最终倍率，来自全局连击倍率配置 |
| `ReasonText` | HUD 展示原因 |

## 开发阶段

### 阶段 1：文档与配置冻结

- 冻结普通卡“攻击即触发”的新规则。
- 冻结连携卡只做少数机制卡，不做所有卡泛化。
- 产出正向连携、反向连携、断链释放的配置说明。
- 明确本阶段不做三连携和终结技。

### 阶段 2：数据结构扩展

- 在 `FCombatCardConfig` 中增加 `LinkConfig`。
- 保留旧字段兼容已有资产，但文档标记为旧语义。
- 增加卡牌标签或邻接条件，让连携卡能判断前后卡类型。
- 增加配置校验，提示 Link 卡缺少 Forward / Backward / Break 配置。

### 阶段 3：卡组解析逻辑

- 普通卡取消动作匹配激活判断。
- `BaseFlow` 改为所有有效出牌都触发。
- 增加正向连携读取 `LastResolvedCard`。
- 增加反向连携写入 `PendingLinkContext`。
- 增加断链清理：战斗系统通知退出连招状态、洗牌、换武器、下一张不满足。

### 阶段 4：BuffFlow 上下文

- Flow 执行时传入卡牌实例、连携来源、连携方向、全局倍率、断链原因。
- 支持同一个 Flow 根据上下文读取前一张/后一张卡的信息。
- 反向连携的效果 B 独立配置为 `BackwardEffect.Flow`，不和后一张卡自身 Flow 混在一起。
- 避免把每种卡牌关系都写死在 C++。

### 阶段 5：UI 与验收

- HUD 能显示普通触发、正向连携、反向赋能、断链释放四类结果。
- 连携卡上需要显示“读取前一张”或“赋能后一张”的提示。
- 断链时要有轻量提示，不能像失败惩罚一样打断战斗节奏。

## 验收场景

| 场景 | 预期 |
| --- | --- |
| 普通攻击卡被 Light 命中触发 | 卡牌消耗，执行 `BaseFlow` |
| 普通攻击卡被 Heavy 命中触发 | 卡牌消耗，执行 `BaseFlow`，不因动作不同失效 |
| 连击数提升 | 普通卡读取全局倍率配置并提升效果，但不影响是否触发 |
| 攻击牌 -> 月光 | 月光读取前一张攻击牌，触发正向效果 A |
| 月光 -> 攻击牌 | 月光先正常释放，后一张攻击牌额外触发单独配置的效果 B |
| 月光等待后一张时战斗系统通知退出连招状态 | 月光断链，按断链策略正常打出或执行 BreakFlow |
| 月光等待后一张但下一张不是攻击牌 | 清理等待，不产生无限连携 |
| 洗牌开始时仍有 PendingLink | 清理 PendingLink，记录断链原因 |

## 已确认决策

1. 退出连招状态由战斗系统判断并通知，背包/卡组/符文系统服务于战斗系统，不自行用时间猜测。
2. 普通卡连击倍率使用全局配置。
3. 反向连携效果 B 是单独配置的新效果，建议落到 `BackwardEffect.Flow`。
