# Codex 开发方案 - 背包符文卡组

## 目标

Codex 负责 512 版本中背包、符文、卡组底座、奖励接入、1D 卡组 UI 事件相关开发。不主动改战斗连招实现，只提供战斗同事可调用的接口。

## 开发边界

| 模块 | 是否由 Codex 负责 | 说明 |
| --- | --- | --- |
| CombatDeckComponent | 是 | 卡组运行时核心组件 |
| 卡牌数据结构 | 是 | DataAsset 或基于 RuneDataAsset 扩展 |
| 洗牌装弹逻辑 | 是 | 固定顺序、打空、冷却、回填 |
| 后台背包兼容 | 是 | 旧 BackpackGridComponent 保留后台职责 |
| 卡组 UI 事件 | 是 | 派发事件给 HUD |
| 关卡奖励进入卡组 | 是 | SelectLoot 后写入后台背包和 DeckList |
| 轻重攻击动作实现 | 否 | 由战斗同事负责 |
| 连招终结击判定 | 否 | 由战斗同事负责 |
| DashSave 窗口实现 | 否 | 由战斗同事负责 |

## 推荐开发顺序

1. 梳理现有 `RuneDataAsset`、BuffFlow、BackpackGridComponent、RewardPickup、LootSelectionWidget。
2. 确认卡牌数据是扩展 `RuneDataAsset` 还是新增独立 `CombatCardDataAsset`。
3. 新增 `CombatDeckComponent` 的状态与接口。
4. 实现 `ResolveAttackCard` 的无卡、消耗、匹配、不匹配、打空逻辑。
5. 实现洗牌冷却和固定顺序回填。
6. 接入后台背包读取和奖励写入。
7. 派发 UI 事件，但第一版 UI 表现可以简化。
8. 与战斗同事联调 Light / Heavy / Finisher / DashSave。

## 第一版必须满足

- 卡组不阻止攻击。
- 卡牌打空后进入洗牌。
- 洗牌期间不触发卡牌。
- 洗牌结束后固定顺序回填。
- 动作不匹配仍消耗卡牌，只触发 BaseFlow。
- 选择奖励后不自动打开旧背包。

## 第一版可以暂缓

- 复杂 Link 组合公式。
- 大量 Passive 卡。
- 随机洗牌。
- 旧 2D 背包高级整理界面。
- 完整美术表现。

## 关键联调接口

```cpp
FCombatCardResolveResult ResolveAttackCard(
    ECardRequiredAction ActionType,
    bool bIsComboFinisher,
    bool bFromDashSave
);
```

战斗同事只需要传入攻击事件，不需要知道卡组如何存储。

## 自查清单

- 是否只修改背包、符文、卡组、奖励接入相关文件。
- 是否没有重写战斗连招系统。
- 是否没有删除旧背包存档能力。
- 是否保留了固定顺序洗牌。
- 是否给 UI 留出了事件接口。
