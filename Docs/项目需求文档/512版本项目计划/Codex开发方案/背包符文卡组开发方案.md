# Codex-背包符文卡组组开发方案

## 组定位

本组负责 512 版本的后台背包、符文卡牌数据、1D 卡组运行时、洗牌装弹、奖励进入卡组。这个组是当前我们的主要开发组。

核心目标是让玩家不再围绕旧 2D 背包整理，而是在战斗中通过 1D 卡组顺序获得构筑反馈。

## 负责范围

| 模块 | 说明 |
| --- | --- |
| CombatDeckComponent | 卡组运行时组件，维护 DeckList、ActiveSequence、CurrentIndex、DeckState、ShuffleCooldown |
| 卡牌数据结构 | CardType、BaseFlow、少数连携卡的正向/反向/断链配置，旧 RequiredAction / MatchedFlow / LinkMode 仅做兼容 |
| 洗牌装弹逻辑 | 固定顺序、卡牌打空、默认 1 秒冷却、冷却结束回填 |
| 后台背包兼容 | 旧 BackpackGridComponent 保留存储、存档、奖励接收能力 |
| 奖励进入卡组 | SelectLoot 后写入后台背包，并更新 DeckList |
| 卡组事件派发 | 给 UI 发送消耗、匹配、连携、洗牌开始、洗牌完成事件 |

## 不负责范围

- 不实现 Light / Heavy 攻击动作。
- 不实现连招动画和攻击蒙太奇。
- 不实现 DashSave 的窗口判定。
- 不制作完整 HUD 美术。
- 不重做关卡结构。

## 核心接口

```cpp
FCombatCardResolveResult ResolveAttackCard(
    ECardRequiredAction ActionType,
    bool bIsComboFinisher,
    bool bFromDashSave
);
```

## 开发步骤

1. 梳理现有 `RuneDataAsset`、BuffFlow、BackpackGridComponent、RewardPickup、LootSelectionWidget。
2. 决定卡牌数据是扩展 `RuneDataAsset`，还是新增独立 `CombatCardDataAsset`。
3. 实现 `FCombatCardInstance`，保存运行时 Guid、来源 DA、等级和来源。
4. 实现 `CombatDeckComponent` 的状态字段。
5. 实现 `ResolveAttackCard` 的无卡、消耗、普通释放、正向连携、反向赋能、断链结果。
6. 实现打空后进入 `EmptyShuffling`。
7. 实现洗牌计时，默认 1 秒，但由武器 DA 配置。
8. 实现冷却结束后按 DeckList 固定顺序回填。
9. 接入奖励选择后的后台背包写入和 DeckList 更新。
10. 派发 UI 事件，不让 UI 直接读内部数组。

## 验收标准

- 4 张卡连续攻击 4 次后进入洗牌。
- 洗牌期间玩家仍可攻击，但不触发卡牌。
- 洗牌结束后按固定顺序回填。
- 普通卡被 Light / Heavy 消耗时都正常触发 BaseFlow。
- 连携卡可以配置正向读取前一张、反向赋能后一张、断链释放三类结果。
- 本阶段不验收三连携和终结技。
- 奖励选择后进入后台背包和 DeckList，不自动打开旧背包。
