# 游戏核心系统迭代
基于玩家集市受击到的问题，背包系统与热度系统感很弱，存在感很低，游戏完整循环太繁琐。经过了上次的处理后仍玩法表达处游戏的体验，认为契合程度较弱。对游戏的背包、符文、热度等战斗系统的辅助系统做如下修改


## 系统调整
**战斗系统修改是因为符文系统调整，想要优化战斗逻辑，将两个系统能更简单的结合。让玩家以最少的心智消耗获得最高的体验**
1. 之前的背包系统想要玩家又战斗、又背包，结果来带的太累。背包已经是一个很重的玩法了，让玩家手动处理太过于困难。==希望在后续的过程中，能让玩家尽可能的以战斗为主。背包自动触发，自动排布（当然玩家想要调整顺序也是可行的）==
	1. 战斗的调整需要修改之前的思路：我们想加入的这些设计都过于爽感，那么就应该在处理的过程重偏向于快速的战斗节奏。这次需要调整战斗节奏的变化。
	2. 战斗开放的更加多变，参考鬼泣的连招逻辑。这样玩家就有更多机会，更多选择。
	3. 削弱冲刺，当前冲刺打断的太频繁，导致削弱了连招的存在。让冲刺暂时无法打断前摇。
	4. 加强连招的存在，让连招的最后一击携带额外的效果，这个效果参与卡牌的连携逻辑。且当卡牌连携失败时，连招的打出也会视为之前的连携成功。
		1. 例如：玩家的攻击为轻-轻-重-重。卡牌需求为：轻-轻-轻-重。当玩家只要最后的连击触发卡牌时，也是为玩家打出了卡牌连携。
	5. 可能需要调整触发按键（待定）。
2. 背包系统之前是2D，想要考虑到平移、旋转、裁剪符文等，功能太多。我们还是修改为1D的序列背包，变成最干净的卡牌构筑逻辑，玩家工具顺序释放卡牌。
	 以下是系统方案
	1. ==卡牌：==玩家攻击就会出牌（卡牌思路）。每次攻击都会使用一张卡牌，默认会携带一些基础卡牌，类似于杀戮尖塔中的“打击”。为了简化战斗的心智，这些会使用的卡牌只考虑为攻击卡牌，如果有防御类卡可作为被动（常驻）、技能（玩家技能按键）等释放。
	2. ==卡牌的释放条件：==卡牌会有两种释放途径：轻攻击、重攻击。如果到了这个卡牌，玩家攻击时做的是不是这个动作（假设需要的攻击动作为轻攻击），卡牌会触发但不会触发连携效果，也不会有倍率奖励。
	   玩家按照卡牌的要求动作释放会触发额外的奖励（暂时通常设置为数值提升）。
	3. ==卡牌具有连携：==类似于noita逻辑，玩家攻击时会遇到连锁卡牌，卡牌将会对攻击后的状态进行叠加。
		例如玩家有两张卡牌：攻击牌、月光，月光牌作为一张连携牌（连携可以读取前一张效果，反转连携可以将效果转递给后一张）。
		1. 当玩家攻击1攻击时，触发攻击牌的伤害特性。
		2. 当玩家攻击2时，玩家会发射月光（发出一道月光，对路径的敌人造成伤害）。
			1. 月光连携效果读取到了前一张卡牌（攻击牌），所以这个时候月光牌发出的效果应该是发出一刀加成伤害的月光。
	4. ==连携长度：==玩家可以主动打造连携，可以为每一个卡牌配上连携词条时就可以连续触发。
	5. ==连携的必要条件：==玩家的连携想要考虑到攻击行为：连携必须符合卡牌的释放要求（即符合卡牌的释放条件）当不符合时这张卡牌无法正常触发就当然失效了。
	6. ==连携中断：==玩家在战斗过程中通常会攻击动作不符，导致连携中断。这里分为两种情况
		1. 玩家攻击动作不符，第二张卡牌连携中断断后连携自然取消。
		2. 玩家冲刺、闪避等动作，动作系统中冲刺保留一次当前的连携 *（参考冲刺连击保留的动作设计）*，如果连击被保留，那么卡牌的连携也将会被保留。
		3. 简单来说卡牌是否能连携很大程度上也取决与玩家是否攻击打出连击。
	7. ==玩家的连招：==玩家动作具有连招，例如鬼泣等格斗游戏的连招。玩家的每个连招动作都会有特殊的连招奖励（详见动作系统调整）。如果触发了连招奖励将会触发额外倍率的奖励。
		1. **例如玩家当前连招为：** 轻-轻-重。卡牌连携的需求也是轻-轻-重。当两者重合时，玩家的第三刀为连招末端的招式，将会对卡牌的连携触发提供一个更高倍率的提升。
		2. **玩家的连招终结技的打造：** 玩家可以打造终结技卡牌。这张卡牌为硬需求，玩家必须根据卡牌的需要打出连携，且连携符合连招。当达到这个效果时，玩家在攻击结束后会主动打出终结技。
3. 关卡问题在于玩家较少会主动的通关关卡后拾取关卡的奖励，类似黑帝斯的设计在这里好像并不合适，有一种玩家通过后制作者主动的施舍。考虑将关卡奖励机制类似为：
	1. 小怪掉落金币（暂定）；
	2. 每个关卡有首领（精英关即可）击杀后可以直接开门去下一关；
	3. 关卡清空的奖励。
	将玩家的奖励变为自己主动去获取的状态。参考网游中的奖励机制，默认就是奖励，不愿意实际上就是变相的削弱，降低玩家抵触情绪。
### 战斗系统制作方案

#### 1. 设计目标

本次战斗系统调整的核心目标是：让玩家的主要注意力回到“快速战斗、轻重攻击选择、连招终结、卡牌连携爆发”上，而不是在战斗外频繁整理背包。

战斗节奏应从原本偏“背包激活 + 热度扩区”的构筑体验，调整为：

> 快速动作战斗 + 轻/重攻击序列构筑 + 周期性卡牌爆发窗口

第一版不追求完整鬼泣式复杂连招树，而是先建立最小可验证规则：玩家通过轻攻击、重攻击、冲刺衔接、连招终结击，消耗 1D 卡组中的卡牌，并根据卡牌要求触发基础效果、连携效果、终结技效果。

#### 2. 战斗基础循环

玩家进入战斗后，任何时候都可以正常轻攻、重攻、冲刺和闪避。卡牌系统不会阻止玩家攻击，只是在玩家攻击时尝试附加额外效果。

最小循环如下：

1. 玩家按下轻攻击或重攻击。
2. 攻击动作正常播放，造成武器基础伤害。
3. CombatDeckComponent 检查当前 1D UI 序列中是否还有可用卡牌。
4. 如果有卡牌，则消耗当前卡牌。
5. 如果玩家动作符合卡牌需求，触发基础效果、匹配奖励、连携资格。
6. 如果玩家动作不符合卡牌需求，卡牌仍然消耗，只触发基础效果，不触发连携和倍率奖励。
7. 如果当前没有可用卡牌，攻击照常执行，但不触发任何卡牌效果。

这条规则必须保证玩家不会因为“没有牌”而无法攻击。卡牌是额外火力窗口，不是攻击许可。

#### 3. 卡牌释放条件

每张攻击卡牌至少需要声明一个释放需求：

| 字段 | 说明 |
|---|---|
| RequiredAction | Light / Heavy / Any |
| BaseFlow | 卡牌被消耗时默认触发的基础效果 |
| MatchedFlow | 玩家动作符合 RequiredAction 时触发的奖励效果 |
| LinkMode | None / ReadPrevious / PassToNext |
| FinisherRequirement | 是否要求连招终结击 |

动作匹配规则：

- 当前卡要求 Light，玩家打 Light：匹配成功。
- 当前卡要求 Light，玩家打 Heavy：卡牌消耗，基础效果触发，连携失败。
- 当前卡要求 Any：轻重攻击都视为匹配。
- 当前无卡：攻击只走武器基础逻辑。

#### 4. 洗牌 / 装弹逻辑

卡组不是无限触发。玩家的卡组数量有限，当卡牌打完后进入洗牌冷却。这个逻辑类似射击游戏中的装弹。

规则如下：

1. DeckList 按玩家当前卡组顺序固定排列，不随机洗牌。
2. ActiveSequence 是当前显示在 1D UI 上、可被攻击消耗的卡牌序列。
3. 玩家每次攻击最多消耗 1 张 ActiveSequence 中的卡牌。
4. 最后一张卡牌被消耗后，立刻进入 EmptyShuffling 状态。
5. EmptyShuffling 状态下，1D UI 暂时清空。
6. 洗牌期间玩家仍然可以攻击，但不会触发卡牌效果。
7. 洗牌冷却结束后，系统立刻按 DeckList 的固定顺序重新填充 ActiveSequence。
8. 下一次攻击即可继续触发卡牌。

洗牌时间默认 1 秒，但不写死在组件中，最终由武器 DA 配置。不同武器可以有不同的“装弹时间”，用于形成节奏差异。

#### 5. 连招与终结击

现有轻攻击、重攻击连招系统继续保留。第一版只要求支持轻/重攻击序列，不需要扩展为完整自由派生树。

连招终结击承担三个职责：

1. 提供比普通攻击更高的打击感和视觉反馈。
2. 作为卡牌连携的高价值判定点。
3. 作为终结技卡牌的触发条件。

示例：

- 玩家动作序列：Light → Light → Heavy。
- 卡牌需求序列：Light → Light → Heavy。
- 第三击同时是 Heavy 连招终结击。
- 此时触发卡牌连携成功 + 终结击倍率 + 终结技卡牌效果。

如果卡牌连携中途失败，但玩家成功打出连招终结击，可允许终结击视为一次“补偿性连携完成”。这个规则用于保护动作玩家的正反馈：玩家只要打出了漂亮的连招，系统就应该给出奖励，而不是只惩罚卡牌顺序错误。

#### 6. 冲刺调整

当前冲刺打断过于频繁，会削弱连招存在感。本次调整建议：

- 冲刺不再允许随时打断攻击前摇。
- 冲刺可以在后摇、桥接窗口、DashSave 窗口中衔接。
- DashSave 保留当前连招进度，同时保留一次卡牌连携上下文。
- 如果玩家在 DashSave 窗口内冲刺，下一次攻击仍可延续连招与卡牌连携。
- 如果玩家在非窗口冲刺，则视为普通闪避，不保留卡牌连携。

该调整的目标不是削弱移动爽感，而是让“攻击承诺”和“连招完成”重新变得重要。

#### 7. 热度系统调整

热度不再作为背包激活区扩展逻辑的前台表达。新定位为：

> 热度强化卡牌连携、终结技表现、链路触发强度。

热度来源优先来自高质量战斗行为：

- 命中敌人。
- 成功打出符合卡牌需求的攻击。
- 连携成功。
- 连招终结击命中。
- 完美闪避。
- 打断敌人前摇。
- 击杀或连锁击杀。

热度提升后，不改变卡牌是否能触发，只改变触发后的收益和表现。例如倍率提升、范围扩大、终结技特效升级、链路提示增强。

#### 8. 验收标准

- 玩家无卡牌时仍能正常攻击、命中、击杀敌人。
- 4 张卡牌连续攻击 4 次后，1D UI 清空并进入 1 秒洗牌冷却。
- 洗牌期间继续攻击不会触发卡牌效果，也不会中断冷却。
- 冷却结束后，1D UI 立即按固定顺序重新显示卡牌。
- 当前卡要求 Light 时，Light 攻击触发匹配奖励，Heavy 攻击只触发基础效果。
- DashSave 窗口内冲刺能保留连招和卡牌连携上下文。
- 非 DashSave 窗口冲刺不能随意打断攻击前摇。

### 背包、符文系统制作方案

#### 1. 设计目标

背包系统不再作为玩家战斗外的主要操作负担。旧 2D 背包系统保留为后台存储、存档、金币、升级、奖励接收的基础设施；前台体验改为 1D 卡牌序列。

玩家需要理解的是：

> 我的攻击会按顺序消耗卡牌；卡牌打完后需要短暂洗牌；拿到新卡后，我的攻击节奏和爆发方式发生变化。

玩家不需要在每场战斗后手动旋转、平移、裁剪、摆放符文。

#### 2. 系统边界

旧 BackpackGridComponent 保留以下职责：

- 持有玩家已获得符文 / 卡牌。
- 负责金币增减、购买、出售等经济逻辑。
- 负责升级、重复卡处理、满级过滤等成长逻辑。
- 负责跨关存档兼容。
- 在需要时作为后台数据来源提供给 CombatDeckComponent。

旧 2D 背包 UI 不作为默认战斗后流程打开。后续可以只在营地、祭坛、特殊事件房中作为高级整理界面出现。

新增 CombatDeckComponent 承担以下职责：

- 维护玩家当前战斗卡组。
- 维护 1D UI 当前显示序列。
- 处理攻击时消耗卡牌。
- 处理动作匹配、连携、终结技需求。
- 处理卡组打空后的洗牌冷却。
- 将卡牌效果交给 BuffFlow 执行。

#### 3. CombatDeckComponent 运行时结构

建议新增组件：`UCombatDeckComponent`。

核心运行时数据：

| 字段 | 说明 |
|---|---|
| DeckList | 玩家当前卡组，按固定顺序排列 |
| ActiveSequence | 当前 1D UI 中可消耗的卡牌 |
| CurrentIndex | 下一张要消耗的卡牌索引 |
| DeckState | Ready / EmptyShuffling |
| ShuffleCooldown | 当前洗牌剩余时间 |
| LastResolvedCard | 上一张成功结算的卡牌，用于连携读取 |
| PendingLinkContext | 当前连携上下文，可被 DashSave 暂存 |

基础流程：

1. 战斗开始或洗牌完成时，将 DeckList 复制到 ActiveSequence。
2. CurrentIndex 从 0 开始。
3. 每次攻击请求 ResolveCurrentCard。
4. 如果 DeckState 不是 Ready，返回“无卡牌效果”。
5. 如果 CurrentIndex 越界，返回“无卡牌效果”。
6. 消耗当前卡牌并推进 CurrentIndex。
7. 如果 CurrentIndex 到达 ActiveSequence 末尾，进入 EmptyShuffling。
8. EmptyShuffling 结束后重新装入 DeckList。

#### 4. 卡牌数据结构

可以基于现有 URuneDataAsset 扩展，也可以新增 UCombatCardDataAsset。为了降低迁移风险，建议第一版新增独立卡牌 DA，但允许它引用现有 RuneDataAsset / FlowAsset。

建议字段：

| 字段 | 说明 |
|---|---|
| CardName | 卡牌显示名 |
| CardIcon | 1D UI 图标 |
| CardType | Attack / Link / Finisher / Passive |
| RequiredAction | Light / Heavy / Any |
| BaseFlow | 卡牌被消耗时触发 |
| MatchedFlow | 动作匹配时触发 |
| LinkMode | None / ReadPrevious / PassToNext |
| FinisherRequirement | 是否要求连招终结击 |
| HeatScalingRule | 热度对该卡的加成方式 |
| Description | 给玩家看的描述 |

卡牌类型说明：

- Attack：基础攻击牌，最适合作为默认卡组。
- Link：连携牌，读取上一张或影响下一张。
- Finisher：终结技牌，要求玩家在指定序列或连招末端触发。
- Passive：不进入攻击序列，作为常驻效果或后台效果。

#### 5. 1D UI 表现

1D UI 是新系统能否被玩家理解的关键。它需要显示：

- 当前卡牌序列。
- 下一张即将消耗的卡牌。
- 当前卡牌要求 Light / Heavy / Any。
- 动作匹配成功时的反馈。
- 连携成功 / 失败原因。
- 洗牌冷却状态。
- 链路系统触发时的原因说明。

洗牌期间 UI 表现：

- ActiveSequence 清空或变暗。
- 显示“洗牌中 / 装填中”进度。
- 进度条时长读取武器 DA 的 ShuffleCooldown。
- 冷却结束后卡牌按固定顺序回到 UI。

链路触发提示示例：

- “月光读取上一张攻击牌：伤害提升”
- “反转连携：本张效果传递给下一张”
- “热度强化：终结技范围扩大”

这类提示必须短、明确、只解释当前发生的事情，不写成长篇教程。

#### 6. 武器 DA 配置

武器需要成为卡组节奏的重要来源。建议在武器 DA 中增加：

| 字段 | 说明 |
|---|---|
| InitialDeck | 初始卡组列表 |
| ShuffleCooldown | 洗牌时间，默认 1 秒 |
| MaxVisibleCards | 1D UI 最大展示张数 |
| bUseFixedDeckOrder | 第一版固定为 true |

不同武器可以通过卡组长度和洗牌时间形成差异：

- 短卡组 + 短洗牌：高频爆发。
- 长卡组 + 长洗牌：规划感更强。
- 重武器：卡少但单卡强。
- 快武器：卡多但单卡弱。

#### 7. 链路系统定位

链路系统不作为首局核心规则，不要求玩家一开始理解。

它作为高级奖励、隐藏效果、进阶构筑存在：

- 玩家获得特定 Link 卡后，才开始接触链路。
- 链路触发时必须有明确 UI 提醒。
- 链路效果需要能用一句话解释。
- 链路不要要求玩家回到 2D 背包里摆格子才能理解。

旧 Producer / Consumer / ChainDirection 逻辑可以保留为后台兼容，但前台表达应转换为“卡牌相邻、前后读取、效果传递”。

#### 8. 验收标准

- 获得新卡牌后，可以加入 DeckList 并在下一轮 ActiveSequence 中出现。
- 旧背包不自动打开，玩家仍能获得奖励并进入卡组。
- 洗牌冷却时间可由武器 DA 调整。
- 固定顺序卡组在每次洗牌后保持同样顺序。
- Link 卡触发时出现明确 UI 提示。
- Passive 卡不进入攻击序列，但能作为后台效果存在。

### 关卡系统制作方案

#### 1. 设计目标

关卡系统需要服务新的战斗卡组循环。玩家每个房间的目标应变成：

> 进入房间 → 快速战斗 → 消耗一轮或多轮卡组 → 清场获得主奖励 → 选择新卡 / 新符文 → 下一房间立刻验证。

关卡不再以“打完后进入复杂整理阶段”为主要节奏。奖励可以被玩家主动拾取，但系统不强迫玩家打开背包处理。

#### 2. 房间节奏

普通房目标时长：45-90 秒。

推荐结构：

1. 进房间后 1-2 秒内开始刷怪。
2. 第一波敌人让玩家快速进入攻击卡组循环。
3. 第二波加入更多敌人密度或位置压力，促使玩家触发连携。
4. 第三波或精英敌人用于测试连招终结、洗牌空窗、热度强化。
5. 清场后生成主奖励拾取物并开启传送门。

每个普通房建议 1-3 波敌人，不建议拉长到 4 波以上。卡组系统需要高频验证，房间过长会让玩家更关注生存消耗，削弱构筑反馈。

#### 3. 敌人职责

普通敌人：

- 负责让玩家爽快触发攻击卡牌。
- 血量不宜过高。
- 适合成群出现，让 Link、范围、弹射、状态传播有表现空间。

精英敌人：

- 负责检验连招、闪避、终结技。
- 有更清晰的前摇和反击窗口。
- 可以迫使玩家经历一次“卡组打空 → 洗牌空窗 → 继续基础攻击 → 卡组回填”的节奏。

Boss / 小 Boss：

- 可以更强调读招和卡组规划。
- 允许玩家经历多轮洗牌。
- 适合作为 Finisher 卡和高热度连携的展示场。

#### 4. 奖励逻辑

第一版保留“清场主奖励”作为主流程：

1. 小怪死亡掉落金币或小资源。
2. 小资源尽量自动吸附，减少拾取压力。
3. 清场后生成主奖励拾取物。
4. 玩家靠近拾取后弹出三选一卡牌 / 符文选择。
5. 选择后奖励自动进入后台背包与 CombatDeckComponent。
6. 不自动打开旧背包 UI。
7. 传送门在清场后开启，玩家可以不拾取奖励直接进入下一关，但这等价于主动放弃成长。

这个设计保留“奖励是玩家主动获取”的感觉，同时避免把奖励流程变成强迫停顿。

#### 5. 房间类型建议

普通战斗房：

- 主要产出基础卡、攻击卡、少量 Link 卡。
- 用于验证核心循环。

精英房：

- 产出更强 Link 卡、Finisher 卡或热度强化卡。
- 敌人设计更强调前摇、霸体、反击。

事件房 / 祭坛房：

- 可以允许玩家调整 1D 卡组顺序。
- 可以提供删卡、升级卡、替换卡等深度构筑操作。
- 这里才适合打开更复杂的管理界面。

商店房：

- 使用金币购买卡牌或被动符文。
- 购买后直接加入 DeckList 或后台 Passive。

Boss 房：

- 检验当前卡组强度。
- 产出高价值 Finisher 卡、武器卡组参数变化、特殊链路奖励。

#### 6. 与现有关卡系统的衔接

现有系统可复用：

- `YogGameMode` 的波次刷怪和清场检测。
- `RewardPickup` 的三选一奖励拾取。
- `LootSelectionWidget` 的奖励选择 UI。
- `Portal` 的传送门开启和切关。
- `LevelFlow` 的关卡事件编排。
- `FindLootSpawnLocation` 的奖励落点计算。

需要调整：

- SelectLoot 后不再自动打开旧背包。
- 奖励类型从 Rune 扩展为 Card / Rune / Passive。
- 关卡清场奖励进入 CombatDeckComponent。
- 小怪金币掉落尽量自动吸附或简化拾取。
- 房间难度和波次数值向 45-90 秒目标重新调参。

#### 7. 第一阶段内容规模

本次以系统底座为主，不一次性替换所有内容。

建议第一阶段最小内容：

- 1 把测试武器。
- 4 张基础攻击卡。
- 2-3 张 Link 卡。
- 1-2 张 Finisher 卡。
- 1 个普通房。
- 1 个精英房。
- 1 个奖励选择流程。
- 1 套 1D 卡组 UI。

测试目标不是内容量，而是验证玩家是否能理解：

1. 攻击会按顺序出牌。
2. 卡组打空后需要短暂洗牌。
3. 无牌时仍可攻击。
4. 连携成功会有更强奖励。
5. 新奖励会改变下一房间的打法。

#### 8. 验收标准

- 普通房能在 45-90 秒内完成。
- 清场后生成主奖励，传送门开启。
- 玩家可以不拾取奖励直接进门。
- 选择奖励后不打开旧背包，而是自动加入卡组。
- 下一房间 1D UI 能看到新卡。
- 小怪金币或小资源不会打断战斗节奏。
- 精英房能让玩家体验至少一次洗牌空窗。
